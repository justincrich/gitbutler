Seeded artifact for AC-1: the `pending` local_review_assignments row written by the LPR-007 hook.

The hook fires via `but_rules::process_commit_rules(vec![rule], &commit_ctx, &mut db)`
where:
  - rule = WorkspaceRule {
      trigger: Trigger::Commit,
      filters: [Filter::ClaudeCodeSessionId("dev".to_owned())],
      action: Action::RequestReview(RequestReviewAction { reviewer: "rev2".to_owned() }),
    }
  - commit_ctx = CommitContext {
      target: "refs/heads/feat",
      session_id: Some("dev"),
      changed_paths: vec![],
    }

Engine outcome (the row asserted in `local_review_assignments().list_by_target("refs/heads/feat")`):

  LocalReviewAssignment {
      id: "<uuid v4>",
      target: "refs/heads/feat",
      reviewer_principal: "rev2",
      state: "pending",            // <-- but_authz::AssignmentState::Pending.name()
      assigned_at: <chrono::Utc::now().naive_utc()>,
  }

  list_by_target() len == 1   (exactly one row written — engine outcome, not a log)
  written     == 1            (the hook's Ok(usize) return value)

The row's `state` is the SAME `AssignmentState::Pending.name()` literal LPR-003's
`request_review`/`assign_reviewer` writes — reusing the LPR-002 enum, not a
parallel literal. The row is built and moved into `local_review_assignments_mut().upsert(...)`
(the LPR-001 Handle), not a forked writer.

Negative controls (would fail if the hook only logged):
  - if the assignment were created for the wrong reviewer (not "rev2") the row's
    `reviewer_principal` would differ -> the test's eq-assertion would panic
  - if the state literal were wrong (not "pending") the test's eq-assertion on
    `AssignmentState::Pending.name()` would panic
  - if no row were written (log-only stub) `list_by_target()` would return empty
    and `rows.len() == 1` would panic
