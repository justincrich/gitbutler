Seeded artifact for AC-4: drive-only proof — the hook writes assignments only,
the merge gate's data source (verdicts) is structurally untouched.

After `process_commit_rules(vec![rule], &commit_ctx, &mut db)?`:

  local_review_assignments:
    id:        <uuid v4>
    target:    "refs/heads/feat"
    reviewer_principal: "rev2"
    state:     "pending"
    assigned_at: <NaiveDateTime>

  local_review_verdicts:   <-- the merge gate's data source
    (empty)

The merge gate (crates/but-api/src/legacy/merge_gate.rs:185-196 `review_verdicts`)
reads ONLY `local_review_verdicts().list_by_target(target)`. The auto-opened
assignment cannot gate a merge because the merge gate does not consult the
assignments table — by construction.

  `process_commit_rules` signature:
      pub fn process_commit_rules(
          rules: Vec<WorkspaceRule>,
          commit_ctx: &CommitContext,
          db: &mut DbHandle,
      ) -> anyhow::Result<usize>

  - Takes `&mut DbHandle`, NOT `&mut RepoExclusive`. The hook acquires NO repo
    guard, so it cannot deadlock the merge gate's own guard acquisition.
  - Returns `Result<usize>`, never panics. A failed write surfaces as `Err`,
    but the commit that triggered the hook has already landed (post-commit).

Negative controls (would fail if the auto-assignment gated):
  - if `enforce_commit_gate` read `local_review_assignments` it would be
    caught by `cargo test -p but-authz invariant_build_gates`
  - if `enforce_merge_gate` consulted `local_review_assignments`, the
    invariant_build_gates grep would also catch it (the test enforces
    functional `Authority` is the only axis any gate may branch on)
  - if `process_commit_rules` returned Err on the row write, the test's
    `?` propagation would short-circuit before the assertions
