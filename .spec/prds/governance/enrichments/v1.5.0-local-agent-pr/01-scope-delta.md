# Scope Delta — v1.5.0 Local Agent PR / Governed-Review Parity (LPR)

This enrichment is **purely additive** to the v1.3.0 scope (and orthogonal to the v1.4.0 STEER enrichment). Nothing in the existing five groups (AUTHZ · GRPS · GATES · LOOP · MGMT) is removed, narrowed, or re-scoped. The enforcement decision — who may do what, and **whether a change lands** — is unchanged; v1.5.0 adds a **review-drive layer** (local PR object + comment threads + assignment state) that orchestrators read to decide _what to do next_, and which **never feeds the merge gate**.

## The load-bearing invariant (constraint, not feature)

Stated first so no reader mistakes the new tables for a second land-truth:

> **The merge gate reads only `local_review_verdicts` at the current head, and that path is untouched.** `local_review_assignments`, `local_review_comments`, and `local_review_meta` are orchestration metadata. An open assignment, a `changes_requested` state, or an unresolved comment **does not block a merge**. The gate gates (verdict-at-head); the new tables drive (orchestration). This is the safe seam: a forged or empty drive table cannot weaken the land decision, because `review_requirement.rs` never reads it. (`crates/but-api/src/legacy/merge_gate.rs` + `review_requirement.rs`.)

## In scope (NEW — the LPR group)

- **A local PR object (Full).** Three new **additive** `but-db` tables on the existing `SchemaVersion`-style additive migration system (the same mechanism that created `local_review_verdicts` and `forge_reviews`):
  - **`local_review_assignments`** — `target` (branch), `reviewer` (principal), `state` (`pending` | `approved` | `changes_requested`), `assigned_at`. The local "who is on the hook to review this" — with the reviewer constrained **distinct-from-author** at the write boundary (R22).
  - **`local_review_comments`** — `target`, `author` (principal), `body`, `file`, `line`, `thread_id`, `resolved`, `created_at`. The local review-comment thread — with resolve constrained to the thread author / assigned reviewer / `reviews:write` holder (R22).
  - **`local_review_meta`** — `target`, `key`, `value`, `created_at`, `PRIMARY KEY(target, key)`. A small per-target structured-metadata row; it carries the **opener principal** (`key = "opener_principal"`) from which the agent-PR tag is derived — deliberately **not** an attacker-influenceable comment body (R23 ≠ R20).
  - The **PR lifecycle state is derived**, not stored as a fourth truth — computed from commits + verdict-at-head (`local_review_verdicts`) + open assignments — mirroring `ForgeReview` fields (`number`/`title`/`body`/`author`/`source_branch`/`target_branch`/`sha`/`labels`/`draft`/timestamps) where a local analogue makes sense.
- **A `but review` command/API surface.** Request/assign a reviewer (`but review request` / `but review assign`), post/list/resolve comments (`but review comment` / `but review comments` / `but review resolve`), and query review status (`but review status`). Built on the **already-shipped** authorities — `Authority::PullRequestsWrite` (open a local review: `but review request`), `Authority::ReviewsWrite` (assignment + the existing verdict path: `but review assign`/`approve`), and `Authority::CommentsWrite` (comments) — **no new authority is introduced**. Drive-layer integrity lives at the `but-api` boundary (assign distinct-from-author; resolve resolver-identity). Like the existing `approve_review`, these are **local-cache writes**: no network, no `DryRun` guard (correct — they touch the local DB, not refs/objects/oplog).
- **A per-project "keep PRs local" setting.** A **per-project operator preference** on `gitbutler_project::Project` (the same class as `forge_override`/`preferred_forge_user`, persisted in the project store and set via the desktop project-settings surface) that makes **agent-authored reviews default to the local layer** — no remote GitHub PR is opened. It is **NOT** a ref-pinned `.gitbutler/{permissions,gates}.toml` governed-config change and **NOT** `administration:write`-gated — it is a trusted-desktop preference under the R12 model (an untrusted project-store write that flips it is the named residual R21). Remote mirroring is **gated behind this setting** (and is itself out of scope to _build_ — see below). When unset/false, the existing remote-PR behavior is unchanged.
- **An automatic agent-PR tag.** A review/PR object created by an **agent principal** (resolved from `BUT_AGENT_HANDLE`) is automatically tagged **`agent-authored`** — a label, with `ForgeReview.labels` as the precedent for a label field, sourced from the dedicated `local_review_meta` opener row (not a comment body) — so agent PRs are programmatically distinguishable from human PRs without a role name in enforcement code.
- **A `but-*` skill contract.** The `but-*` skills **auto-set these PRs local** when used — i.e. the skill workflow invokes the local `but review` surface with the project "keep PRs local" posture, so an agent driven by a `but-*` skill produces a local PR by default. This is the **capability/contract the skills depend on** (the skills assume the local surface exists and is the default), documented as the skill-facing guarantee.
- **A reconciler usage model + the auto-hook.** The orchestrator-as-reconciler pattern is documented as the intended consumption model: **assignment → dispatch the assigned reviewer; unresolved comment → dispatch remediation; approved-verdict-at-head → merge.** Every decision is a projection of `but` review state, not a private orchestrator state machine. The existing **`but-rules` (Trigger → Filter → Action) engine** is named as the home for an auto **"review-requested"** hook on commit (trigger: commit on a watched branch; filter: principal/branch; action: open a local review assignment) — **extending** the shipped rules engine with a new commit `Trigger` variant + a review-assignment `Action` variant (today `but-rules` `Trigger` = FileSytemChange/ClaudeCodeHook and `Action` = Assign/Amend/NewCommit — neither covers commit→open-assignment, so this is an extension, not a drop-in).

## Preserved from sprints 1–8 + STEER (explicit)

The enrichment **depends on and preserves** every shipped surface:

- **The merge gate's land-truth** — `local_review_verdicts` read at head, `min_approvals`, `require_distinct_from_author`, required-group satisfaction (`merge_gate.rs` + `review_requirement.rs`, Sprint 01b/04) — **untouched**. v1.5.0 reads _around_ it, never _into_ it. (The drive layer mirrors `require_distinct_from_author` for `assign_reviewer` at its own boundary — R22 — without touching the gate's copy.)
- **The authorities** — `Authority::PullRequestsWrite` + `Authority::ReviewsWrite` + `Authority::CommentsWrite` (Sprint 01a/02) are **reused as-is**; no new authority is added, and the `but review` surface is permission-checked at the `but-api` boundary like every other governed action (so a principal without `pull_requests:write` cannot open a review, without `reviews:write` cannot assign, and without `comments:write` cannot comment).
- **The DB + additive-migration mechanism** — per-project SQLite (`.git/gitbutler/but.sqlite`), `SchemaVersion`-style additive migrations (Sprint 01a) — the three new tables are _additive_ (a forward-only migration adding tables, no change to existing tables).
- **The `forge_reviews` shape** — the remote-PR cache (Sprint sync path) is the **parity reference** the local PR object mirrors, and the **bridge anchor** for the deferred remote-mirroring path.
- **`approve_review`** — the local verdict write (Sprint 01b) is reused unchanged.
- **The `but-rules` engine** — Trigger → Filter → Action (the rules surface Sprint 06b exposes) is the home for the auto "review-requested" hook; v1.5.0 adds a hook _to_ it, it does not re-build it.
- **The CLI surface conventions** — `but perm` / `but group` (Sprint 05) and `but perm list` (Sprint 05) establish the `but-api`-boundary-checked verb pattern the new `but review` verbs follow.
- **The STEER enrichment (v1.4.0)** — orthogonal and compatible: a `but review`-surface denial is itself an actor-correctable denial that STEER can decorate with `authorized_actions` (e.g. surfacing `but review status`), but LPR introduces **no dependency** on STEER and STEER introduces none on LPR.

Where v1.5.0 implies a change to any shipped surface, it is recorded as a delta to apply when the freeze lifts — never edited into a frozen sprint now. This enrichment lands as **Sprint 07 (LPR)** (the human-directed slot; the existing STEER sprint is renumbered 07→08 — see [05-delta-replan.md](./05-delta-replan.md) §N1).
