# Functional-Permission Agent Governance for GitButler — Overview

## Product Description

This initiative turns **GitButler into a commit/merge policy-enforcement layer for orchestrated multi-agent coding**. It introduces a **functional, GitHub-mirrored permission system** in GitButler and adds **principal grouping** (grant functional permissions to users _or_ groups), then wires those permissions into **two thin gates on GitButler's own actions** — a **commit gate** (functional permission + branch protection) and a **merge gate** (merge authority + a configurable review requirement at head). The result: a discrete agentic actor that drives GitButler is held to the **same gates — review and permission — as a human**, and the gates are unopinionated mechanism (a functional permission resolver + a config-driven clause set), with every opinion living in committed, ref-pinned config (`.gitbutler/permissions.toml`, `.gitbutler/gates.toml`). The feature is **native to GitButler and grounded in GitButler's own crates** — a new `but-authz` crate plus `but-api`, `but-workspace`, `but-db`, and the `but` CLI verb modules (`crates/but/src/args/`; `but-clap` is a CLI-**docs** generator, not the verb surface).

The enforcement surface is **GitButler's action set only** (`but` CLI / `but-api` / Tauri / N-API). An agent may edit files in the **shared working directory** freely — GitButler's model is **virtual branches over one working tree, not a worktree per branch** — and nothing it writes matters until it becomes a **GitButler git action** (`commit`, `integrate`/`merge`, `push`, `review`, `comment`). Those are what get permission-checked and gate-checked, and they are gated **whichever branching mechanism produced them** — virtual branches/stacks (the default), normal-git single-branch, or opt-in worktrees. The seam is effective only for callers that route through `but-api`; the N-API (Electron lite) surface must be audited so it cannot reach lower-level crates ungoverned (R14). Role separation (implementer vs reviewer vs maintainer) **emerges from the functional permission set** — never from a role name in code.

## The thesis: irrigation, not a dam

> **The best way to harness the power of water isn't to stop its flow — it's to redirect it.** Dams and irrigation channels don't fight the river; they shape where it goes. Functional permissions are that irrigation system for agents.

An agent driving GitButler _can_, in principle, break out of the governed path (the fence around raw git is an accepted, leaky guardrail in this slice — see [01-scope.md](./01-scope.md), not a steel trap). The bet is not that escape is impossible — it is that **escape is more expensive than compliance.** If getting a review and writing good code is the lowest-effort way to make progress, a goal-directed agent flows down that channel and _follows Butler's governance_. We make the governed path the path of least resistance, and the agent's own optimization does the rest. We do not cage the agent; we **grade the riverbed** so the water runs toward good code.

This reframes the usual "but hooks are bypassable" objection from a weakness into the design's center of gravity: we are not trying to make defection impossible — we are trying to make compliance _cheaper than defection_, and to hold every actor (human or agent) to the same legible gates when they reach the only acts with consequence.

## Problem Statement

The place teams _actually run_ parallel agents — a rich local git client with a human in the loop — has governance that is advisory at exactly the point that matters. Three structural gaps:

1. **GitButler has no enforcement boundary.** It is a frictionless client: virtual branches in one working tree, hunks auto-assigned, the app committing directly, and landing happens as a _remote forge_ operation (the trunk ref is immutable locally). The only act with consequence (landing) crosses no gate GitButler owns. An agent driving GitButler is bound by nothing GitButler enforces — it is asked to honor a loop, not held to one.

2. **There is no functional permission model over principals.** GitButler's existing `Permission`/`_with_perm` is a **repository-access lock** (concurrency — the right to exclusively touch the working tree), not principal authorization. Nothing answers _may this principal perform this action_. There is no notion of an implementer that can commit but not merge, a reviewer whose edits are inert, or a human maintainer whose approval the gate requires — and no way to express any of that as committed config.

3. **Discrete agents are not held to human gates.** A human contributor goes through review and branch protection. An agent, today, does not — it commits and merges on the same footing as the tool's owner. There is no mechanism that makes "an agent's code must clear an AI code-level review _and_ a human feature-level approval before it lands" a property of the system rather than a hope.

## Solution Summary

**Govern the git action, not the tool and not the agent's reasoning.** This initiative builds a functional permission layer and two gates over GitButler's action surface, plus a human surface to manage the governance config. Five functional groups:

- **Functional permissions mirroring GitHub** (AUTHZ): a per-principal **AuthoritySet** (`contents:write`, `reviews:write`, `merge`, `comments:write`, `administration:write`, …) is the authoritative config in committed, ref-pinned `.gitbutler/permissions.toml`; named roles are optional load-time sugar that desugar to functional sets; superuser = all. Every consequential GitButler action checks the principal's functional permission; a miss (or an unknown principal, or unreadable config) **fails closed** and returns the structured `{code, message, remediation_hint}` + exit 1. **No enforcement path keys off a role name.** A new `but-authz` crate hosts it; the `but-api` boundary acquires authority near the wrapper (mirroring the existing `_with_perm` composition for local actions; a pre-call guard for async forge actions), without overloading that lock type. The seam binds only callers that route through `but-api` — the N-API surface is audited so it cannot bypass it (R14).

- **Principal grouping** (GRPS): functional permissions are granted to **users or groups**; a principal's effective AuthoritySet is the **union** of its own grants and its groups' grants (the GitHub teams model). Group membership and grants are themselves committed, ref-pinned governed config — read at the target ref — so an agent cannot add itself to a privileged group on its own change and self-escalate. `but group` manages create/grant/membership and group delete.

- **The two gates** (GATES): the **commit gate** (at GitButler's commit narrow-waist, applied across every branching mechanism) checks `contents:write` + branch protection — no direct commit to a protected branch. The **merge gate** (at GitButler's PR-merge action) checks `merge` authority + a configurable review requirement: ≥1 approving review, distinct from the author, from each required group, valid at the current head. The engine is **unopinionated**; every threshold and required-group lives in `.gitbutler/gates.toml`. The merge gate's review requirement reads the local `local_review_verdicts` store, which is **forgeable by direct DB write (R6, High)** — sound only for reviews submitted through the governed `but review` action (an accepted-leak honesty item, same class as the fence).

- **Governed orchestration loop** (LOOP): a demonstration that ties it together — an implementer agent (`contents:write`, no `merge`), a reviewer agent (`reviews:write`), and a human maintainer (the `merge` holder, final feature-level approver) run implement→review→merge, with **role separation emerging purely from functional permissions**. The loop's "open a PR" / "submit a review" steps gate on the governed `but pr` / `but review` actions. "Human at the feature level, AI at the code level" is expressed as three lines of `gates.toml` (`require_approval_from_group = ["code-reviewers", "maintainers"]`), not as code that knows what a human or an AI is.

- **Governance management UI** (MGMT): a human surface in the **SvelteKit/Tauri desktop app** (`apps/desktop`) where an admin manages — per agent — functional permissions, group membership, branch gates, and per-agent automation rules. It is a **governed front-end** over `but-authz` (every write goes through the same admin-gated, ref-pinned, committed path via `but-api`→Tauri→`but-sdk`; pending-until-committed; read-only without `administration:write`; no self-escalation), extending GitButler's existing settings + `rules/` patterns with no new design-system work — **not a bypass**, and not a new _agent_ surface (agent-facing enforcement stays CLI/action-boundary).

The bet: hold discrete agentic actors to the same review/permission gates as humans by making GitButler enforce functional permissions on its own git actions — and make the governed path the cheapest path, so agents flow toward good code instead of being walled away from bad code.

> **Prior art (concept only — no dependency).** The functional-permission + gated-loop _concept_ was first prototyped in a separate project (Spoke). This PRD is an **independent, native GitButler implementation** grounded entirely in GitButler's own crates — it borrows the concept and may adapt logic, but integrates **no Spoke package**, depends on **no Spoke service**, and is referenced against GitButler source, not Spoke. Deferred hardening layers (named, not denied — see [01-scope.md](./01-scope.md)): an HMAC integrity check on the review store (closing R6's forgeability) → Ed25519-signed reviews & comments, a full multi-clause gate, auto-run validation feeding the gate, an audited break-glass override, and the steel-trap transport boundary.
