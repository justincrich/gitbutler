# Functional Groups

| Group                                   | Prefix | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------------------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Functional Permission System            | AUTHZ  | Permissions are **functional, not role-based**: a per-principal `AuthoritySet` (`contents:write`, `reviews:write`, `merge`, …) in committed, ref-pinned `.gitbutler/permissions.toml` is the authoritative config; named roles (read/triage/write/maintain/admin) are optional presets that desugar to functional sets at load; superuser = all; a raw functional list loads without a role. A new `but-authz` crate owns `Authority`/`AuthoritySet`/`authorize()` (named distinctly from GitButler's existing repo-access `Permission` lock). Every consequential GitButler action checks the principal's functional permission and returns the structured agent-readable denial `{code:"perm.denied", message, remediation_hint}` + exit 1 — and **no enforcement path keys off a role name**. The seam binds only callers that route through `but-api` (Tauri / `but` CLI / TUI / N-API — the N-API surface is audited, R14). Includes identity confinement (`BUT_AGENT_HANDLE`, no `--as` escalation) and config-change authority (`administration:write`).                                              |
| Principal Grouping                      | GRPS   | Functional permissions are granted to **users or groups**. A principal's effective `AuthoritySet` is the **union** of its own grants and the grants of every group it belongs to (the GitHub teams model). Group definitions, grants, and membership are committed, ref-pinned governed config, read at the **target ref** when authorizing — so a change that adds its author to a privileged group is **ineffective** until committed to the target branch (no self-escalation). `but group` (create/grant/add-member/remove-member/delete) is itself gated by `administration:write`. This is the net-new grouping capability beyond a basic per-principal permission model.                                                                                                                                                                                                                                                                                                                                                                                                                              |
| The Two Gates (Control Plane)           | GATES  | Two deterministic gates on GitButler's own action boundaries, applied **branching-mechanism-agnostically** (virtual branches/stacks, normal-git, opt-in worktrees). The **commit gate** (at the commit narrow-waist covering every commit/integrate entry point) checks `contents:write` + branch protection so a direct commit to a protected branch is rejected with `{code:"branch.protected"}` — landing must go through review → merge. The **merge gate** (at GitButler's PR-merge action — the trunk ref is immutable locally) checks `merge` authority **and** a configurable review requirement (≥1 approving review, distinct from author, from each required group, at the current head), returning `{code:"gate.review_required", unmet:[…]}` on a miss. The engine is **unopinionated** — thresholds and protected branches live entirely in committed, ref-pinned `.gitbutler/gates.toml`. The review requirement reads the local `local_review_verdicts` store, which is forgeable by direct DB write (R6, High) — sound only for reviews submitted through the governed `but review` action. |
| Governed Orchestration Loop             | LOOP   | The demonstration that ties it together: an implementer (`contents:write`, no `merge`), a reviewer (`reviews:write`), and a human maintainer (`merge`) run implement→review→merge through GitButler's gated actions, and **role separation emerges purely from the functional permission set** — the implementer is denied the land, the reviewer's edits are inert, only the maintainer lands and only after the configured reviews exist at head. The loop's "open a PR" / "submit a review" steps gate on the governed `but pr` / `but review` actions (each permission-checked). "Human at the feature level, AI at the code level" is expressed as `gates.toml` config (`require_approval_from_group = ["code-reviewers","maintainers"]`), not as code that knows what a human or an AI is.                                                                                                                                                                                                                                                                                                             |
| Governance Management UI                | MGMT   | A human surface in the **SvelteKit/Tauri desktop app** (`apps/desktop`) where an admin manages, per agent, the governance config the other groups enforce — functional permissions, group membership, branch gates, and per-agent automation rules. It is a **governed front-end** (every write goes through the same `but-authz` admin-gated, ref-pinned, committed path via `but-api`→Tauri→`but-sdk`; batch-saved + pending-until-committed; read-only without `administration:write`; no self-escalation) — **not a bypass**. It **extends existing patterns** (a new admin-only Project Settings page + reuse of the existing `rules/` components) with no new design-system work. The agent-facing _enforcement_ stays CLI/action-boundary; MGMT adds the _human management_ surface (with a desktop-CT-harness prerequisite, B14).                                                                                                                                                                                                                                                                    |
| Capability-Aware Denials                | STEER  | Turns each governance denial into a **flow-diverter**: alongside _why it failed_, the denial returns _what this principal CAN do right now_ — a machine-readable `authorized_actions` menu + a `class` recoverability tag + `held_permissions` + `do_not` — so a goal-directed AI agent diverts down the governed path instead of hard-quitting, looping, or bypassing. **Net-additive** (v1.4.0 enrichment): adds fields to the denial payload only; changes no gate decision, no code set, no enforcement path. The menu is derived from a single-source route→Authority table (same table the gate enforces against → no lying menu); text from a closed code-owned catalog (anti-injection); degrades to the vertical handoff when no lateral action exists.                                                                                                                                                                                                                                                                                                                                             |
| Local Agent PR / Governed-Review Parity | LPR    | Gives GitButler's **local** review layer GitHub-PR parity so an orchestrator drives the whole implement→review→merge loop off `but`'s own review state — a **reconciler over `but`, not a private state machine**. Reviewer assignment + a review comment thread + a **derived** PR lifecycle + a per-project "keep PRs local" default + an auto `agent-authored` tag. **Net-additive** (v1.5.0 enrichment): three new `but-db` tables (`local_review_assignments`, `local_review_comments`, `local_review_meta`) + a `but review` verb surface, reusing existing authorities (**no new `Authority`**). The **safe seam**: the new tables drive orchestration and **never feed the merge gate** (the gate still reads only `local_review_verdicts`@head); agent PRs are local by default.                                                                                                                                                                                                                                                                                                                    |

Prefixes AUTHZ / GRPS / GATES / LOOP / MGMT / STEER / LPR are distinct and unambiguous within this PRD. **30 UCs** — above the 3–25 single-PRD budget; this is the intended outcome of folding the two net-additive enrichments (STEER v1.4.0 + LPR v1.5.0) into the live PRD, **not** fresh scope growth — each enrichment is append-only and frozen-aware (no existing group rescoped). Every group has ≥1 UC. AUTHZ establishes the functional permission model (incl. fail-closed default-deny + the N-API seam audit); GRPS adds principal grouping; GATES is the minimal two-gate control plane (review + permission, not a full clause set); LOOP is the integration demonstration; **MGMT** adds the human governance-management surface in the SvelteKit/Tauri desktop app, extending existing settings/rules patterns as a governed front-end; **STEER** turns each denial into a capability-aware flow-diverter (additive payload fields, no gate-decision change); **LPR** adds local-PR / governed-review parity (three additive drive-only tables that never gate). **201 ACs / 205 e2e criteria**, 23 risks — see [11-e2e-testing-criteria.md](./11-e2e-testing-criteria.md) and [`10-technical-requirements/07-technical-risks.md`](./10-technical-requirements/07-technical-risks.md).
