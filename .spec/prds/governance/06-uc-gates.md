# Use Cases: The Two Gates (Control Plane) (GATES)

The control plane is two deterministic checks on GitButler's own action boundaries — no advisory call, no skip. The **commit gate** decides what commits may land on a branch by enforcing the principal's functional `contents:write` plus branch protection, applied at GitButler's commit narrow-waist **across every branching mechanism** (virtual branches/stacks, normal-git, opt-in worktrees). The **merge gate** decides what lands on a protected branch, at GitButler's **PR-merge action**, by checking `merge` authority plus a **configurable** review requirement (≥1 approving review, distinct from the author, from each required group, valid at the current head). The gate engine is **unopinionated**: it ships the mechanism (a permission check + a config-driven review-requirement evaluator) and **no** baked roles or workflow opinions — every threshold and protected branch lives in committed, ref-pinned `.gitbutler/gates.toml`. Permission says _who may attempt_; the gate says _whether the governed action proceeds_ — even a `merge`-holder is bound by the configured requirement. (Source note: GitButler's trunk ref is immutable locally, so the merge gate binds the _governed_ `but` merge action; a merge done entirely on the forge or by raw push is an accepted-leak bypass — see Known Limitations.)

> **Review-store forgeability (R6, High — accepted-leak honesty).** The POC merge gate trusts a **forgeable local store** (`local_review_verdicts`). An agent with DB/filesystem write access can INSERT a forged approving review; the merge gate is then **trivially satisfiable by direct DB write** — the same accepted-leak class as the fence (R1). The merge gate's review requirement is satisfied only by review submissions made through the governed `but review` action; a forged review via direct DB write (R6 / R-NEW-1) is **outside governance scope**, and the LOOP demo (UC-LOOP-01) assumes honest review submission. Mitigation: submit-review through the governed `but` action only. Deferred hardening: an HMAC integrity check keyed by a repo-local admin secret (then Ed25519-signed review artifacts). No section may present the local review store as tamper-proof.

## UC-GATES-01: Commit gate — functional permission + branch protection (mechanism-agnostic)

The merge gate is useless if a principal can commit straight onto a protected branch. The commit gate is GitButler's commit path made enforcing — at the **narrow waist every commit caller funnels through** (`but_workspace::commit_engine::create_commit`), so it covers virtual-branch/stack commits, normal-git commits, **and** the opt-in worktree commit/integrate path identically (a gate wired to one mechanism would be bypassed by another). It resolves the acting principal from `BUT_AGENT_HANDLE`, loads the `AuthoritySet` from the ref-pinned config, and resolves branch protection for the target from committed `.gitbutler/gates.toml`. A principal lacking `contents:write` is denied (its edits never reach a ref); a direct commit to a protected branch is rejected with `{code:"branch.protected"}` — naming the protected branch and the legitimate path (land through a reviewed merge); a feature-branch commit by a `contents:write` principal is accepted.

### Acceptance Criteria

☐ The commit gate accepts a commit to a feature branch/stack by a principal holding `contents:write`, so non-protected work flows normally
☐ Via the governed `but` commit action, the commit gate rejects a direct commit to a protected branch (e.g. `main`) with `{error:{code:"branch.protected", branch, message}}` + exit 1, so a sneak commit cannot land outside the merge gate
☐ The commit gate's `branch.protected` message names the protected branch and tells the principal to land changes through a reviewed merge from a feature branch
☐ The commit gate denies a commit by a principal lacking `contents:write` with the `perm.denied` contract, so a read-only principal's edits never reach a ref
☐ The commit gate applies **identically regardless of branching mechanism** — a virtual-branch/stack commit, a normal-git commit, and an opt-in worktree commit/integrate are each gated at the same narrow-waist, so no mechanism is an ungated path
☐ The commit gate resolves the acting principal from `BUT_AGENT_HANDLE` and its `AuthoritySet` from the ref-pinned `.gitbutler/permissions.toml`, consistent with UC-AUTHZ-02, and fails closed on an unknown principal or unreadable config (UC-AUTHZ-04)
☐ The commit gate resolves branch protection for the target branch from committed `.gitbutler/gates.toml`, rather than a hardcoded branch list
☐ System has a passing integration test against the real GitButler commit path + real git that commits to a feature branch with a `contents:write` principal (asserts accepted) and attempts a direct commit to a protected `main` (asserts rejected with `error.code == "branch.protected"`)
☐ System has a passing integration test that exercises the commit gate via an opt-in worktree (`but worktree new` → commit/integrate) and asserts the same `contents:write` + branch-protection decision as the virtual-branch path

## UC-GATES-02: Merge gate — merge authority + configurable review requirement @head

A merge gate that counts any approving review is not a gate. At GitButler's **PR-merge action** (`but-api` `legacy/forge` — the local trunk ref is immutable, so this is where a governed merge is initiated), the gate checks two things before a change lands on a protected branch: (1) the acting principal holds `merge`, and (2) the **configured** review requirement is satisfied. The requirement lives in `.gitbutler/gates.toml` per target branch — `min_approvals`, `require_distinct_from_author`, `require_approval_from_group[]` — read at the **target ref**, so a change whose head weakens the requirement cannot weaken the gate that judges it. Approvals are evaluated at the **current head**: an approval at an earlier head is dismissed once the head advances, and a self-approval does not count when `require_distinct_from_author` is set. An unmet requirement returns `{code:"gate.review_required", unmet:[…]}`. The engine bakes in none of these opinions — turning a knob in `gates.toml` is the only way to change strictness. (The gate binds the governed `but` merge; a forge auto-merge / UI merge / raw push is an accepted-leak bypass — Known Limitations.)

The approvals the gate counts come from the local `local_review_verdicts` store, which is **not integrity-protected (R6, High)**: a direct DB write can forge an approving verdict, trivially satisfying the requirement — the same accepted-leak class as the fence (R1). The gate's review requirement is therefore sound only for reviews submitted through the governed `but review` action; the LOOP demo (UC-LOOP-01) assumes honest review submission, and a forged review via direct DB write is outside governance scope. Deferred hardening: HMAC/Ed25519 review integrity (see scope Known Limitations).

### Acceptance Criteria

☐ Via the governed `but` merge action, the merge gate blocks a merge to a protected branch by a principal lacking `merge` with the `perm.denied` contract, even when the review requirement is satisfied — permission says who may attempt, the gate says whether it lands
☐ The merge gate blocks a merge when fewer than `min_approvals` approving reviews exist at the current head, returning `{error:{code:"gate.review_required", unmet:[…]}}` that lists the unmet requirement(s)
☐ The merge gate does not count a review by the change's author when `require_distinct_from_author` is set, so a self-approval does not satisfy the requirement
☐ The merge gate dismisses an approval recorded at a prior head once the head advances (so a stale approval does not count, and the reviewer must re-approve), and the `unmet` payload distinguishes `no_approval` from `approval_stale_at_head` so the orchestrator can re-route the right reviewer
☐ The merge gate requires an approving review from each group named in `require_approval_from_group`, so a merge needs an approval from every required group (e.g. both `code-reviewers` and `maintainers`) to proceed
☐ The merge gate reads its requirement from committed `.gitbutler/gates.toml` at the **target ref**, so a change whose head edits the requirement to drop it cannot weaken its own gate
☐ The merge gate fails closed when the target-ref `.gitbutler/gates.toml` is malformed or names a group undefined in the target-ref config, denying rather than treating the requirement as satisfied (consistent with UC-AUTHZ-04)
☐ The merge gate counts a distinct, current-head approval from each required group, with the acting principal holding `merge`, as satisfying the requirement and allows the governed merge to proceed
☐ System has a passing integration test against the real GitButler PR-merge action + real git that opens a PR with no approving review (asserts blocked with `gate.review_required`), adds a self/non-distinct approval (asserts still blocked), then adds a distinct approval from each required group with a `merge`-holding principal (asserts the merge proceeds) — all approvals submitted through the governed `but review` action (the forgeable direct-DB-write path is out of scope per R6 and is NOT under test)
☐ The merge gate NEVER reads `gates.toml` from the working tree or the feature head when deciding — ONLY the target-ref config blob — so a staging-area or uncommitted edit cannot influence the gate decision
