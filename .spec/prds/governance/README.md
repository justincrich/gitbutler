# Functional-Permission Agent Governance for GitButler (POC) · PRD

Introduce a functional, GitHub-mirrored **permission system** in GitButler and add **principal grouping**, then wire both into **two thin gates on GitButler's own actions** (commit + merge), applied **branching-mechanism-agnostically** (virtual branches, normal-git, opt-in worktrees) — so a discrete agentic actor is held to the **same review/permission gates as a human**, the engine is unopinionated (functional permissions + config-driven clauses, never roles), and the governed path is made the _cheapest_ path rather than the only one. Native to GitButler, grounded in its own crates.

> **The thesis — irrigation, not a dam.** You don't harness a river by stopping it; you channel it. Functional permissions are the irrigation system: an agent _can_ break out (the fence is an accepted, leaky guardrail in this slice), but if compliance is cheaper than defection, a goal-directed agent flows toward good code and follows Butler's governance. We grade the riverbed, not cage the water. See [00-overview.md](./00-overview.md).

> **Prior art (concept only — no dependency).** The functional-permission + gated-loop _concept_ was first prototyped in a separate project (Spoke). This PRD is an **independent, native GitButler implementation** grounded entirely in GitButler's own crates — it borrows the concept and may adapt logic, but integrates **no Spoke package**, depends on **no Spoke service**, and is verified against GitButler source. Deferred hardening layers (named, not denied): an HMAC integrity check on the review store (closing R6's forgeability) → Ed25519-signed reviews, a full multi-clause gate, auto-run validation, an audited break-glass override, and the steel-trap transport boundary — see [01-scope.md](./01-scope.md).
