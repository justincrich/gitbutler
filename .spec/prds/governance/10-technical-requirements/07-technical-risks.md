# Technical Risks

Risks carried into implementation, with the mitigation each gets in this slice. The first two are **accepted by design** (the POC's scope line); the rest are correctness risks the build must close.

**Risk register: 14 risks.** R6 is **High** (the merge gate's review store is forgeable by direct DB write); R14 (**Medium**, Accepted residual) covers the N-API / direct-caller bypass of the `but-api` enforcement seam.

| # | Risk | Severity | Why it matters | Mitigation in this slice |
|---|---|---|---|---|
| R1 | **The fence is bypassable** (client git hooks / harness hooks) | Accepted | An agent with a raw shell, `--no-verify`, `core.hooksPath` edits, or git plumbing can land outside the gates | **Accepted, not closed** — the POC's claim is the irrigation claim (cheapest path), not impossibility. Named in `01-scope.md` Known Limitations. Steel-trap (server-side pre-receive / sandbox) is the deferred closure. The build must NOT pretend this is closed. |
| R2 | **Identity rests on injected `BUT_AGENT_HANDLE`** | Accepted (personal-tenant) | `--as` is denied, but the env var can be re-exported by a **sub-process re-exec**, and a direct `but-api`/N-API caller could supply a principal out-of-band | Confinement denies the in-band `--as`; `_with_authz` resolves the principal from the confined source, never a caller-supplied arg (UC-AUTHZ-03); the sub-process/env-reset residual is a named accepted leak (`01-scope.md`). Per-agent key-mint is the deferred hardening. The build must NOT claim full identity confinement. |
| R3 | **`Permission` name collision** — overloading GitButler's repo-access lock | High | Conflating the lock with authorization would corrupt both concurrency and security reasoning | The naming guardrail (`02-system-components.md`): new `Authority`/`AuthoritySet`/`_with_authz`; mirror the `_with_perm` *shape*, never the *type*. Reviewer grep-asserts no overload. |
| R4 | **Ref-pin correctness** — reading config from the head instead of the target ref | High | A change could grant itself authority or weaken its own gate (the exact hole grouping could open) | All governance config (`permissions.toml`, `gates.toml`, group membership) read at the **target ref** (UC-GRPS-02, UC-GATES-02). Integration test asserts a self-escalating head is ineffective. |
| R5 | **A role name leaks into an enforcement path** | High | Would make governance key off a label, breaking the functional-not-role invariant | Every check tests an `Authority`; roles desugar at load and are never seen by enforcement. Grep-assert AC in UC-AUTHZ-02 / UC-LOOP-01. |
| R6 | **Review-state location + integrity** — where the merge gate reads approvals @head, and the forgeable local store | **High** | The gate can't be sound without distinct-approval-at-current-head; and the local record is **not integrity-protected** — the POC merge gate trusts a forgeable local store (`local_review_verdicts`). An agent with DB/filesystem write access can INSERT a forged approving review directly; the merge gate is then **trivially satisfiable by direct DB write**, so the review requirement is only as strong as the assumption that reviews arrive through the governed `but` action | A NEW `but-db` table `local_review_verdicts` (`head_oid` load-bearing; `but-forge-storage` holds only settings and `forge_reviews` is a disposable cache — neither reusable). Tested for stale-head dismissal **via the governed review-submission path** — a direct DB INSERT is untestably forgeable and is NOT the path under test. **Mitigation:** submit-review through the governed `but` action only; a direct DB write is the **same accepted-leak class as the fence (R1)**. **Deferred hardening:** an HMAC integrity check keyed by a repo-local admin secret (then Ed25519-signed review artifacts). The build must NOT present the local review store as tamper-proof. |
| R7 | **Multiple ref-advancing paths** — a mechanism left ungated is a hole | **Blocking** | GitButler has many commit/integrate/push paths (`commit_engine`, `but-workspace::branch::apply`/`unapply`, `but-worktrees::integrate`, the forge PR-merge, push); a gate wired to one is bypassed by another, and the local trunk is immutable (so "land" is a forge action, not a local stack-land) | The build MUST enumerate every commit/ref-advancing entry point and gate or explicitly accept each; a silently-ungated path is a blocking gap, not a planning note (UC-GATES-01/02, architecture Stance 1b). |
| R8 | **Authorization not applied at every consequential action** (a missed route) | Medium | One unwrapped action route is a hole | Acquire authority at the `but-api` wrapper for *every* consequential action (one boundary); a reviewer checklist enumerates the consequential routes and asserts each is `_with_authz`-wrapped. |
| R9 | **Scope creep into the agent runtime** | Low | Drifting into driving agent logic violates Assumption 2 and bloats the POC | The PRD's scope line is explicit; reviewer rejects any runtime/turn-loop/system-prompt work as out of scope. |
| R10 | **Config-bootstrap self-escalation** | High | An `administration:write` holder could rewrite `gates.toml` to unprotect a branch, then escalate `permissions.toml` on it — the ref-pin stops *feature-head* escalation but not same-ref bootstrap by an admin who can also reach the fence | Config-change authority is itself `administration:write`-gated and ref-pinned; the same-ref bootstrap by an admin + fence-bypass is a **named accepted residual**. A bootstrap invariant (un-protecting a branch must itself clear the existing gate) is the hardening follow-up (`01-scope.md`). |
| R11 | **The merge gate binds the governed `but` action only** | Accepted | A merge done entirely on the forge (auto-merge / UI / CI) or by raw push to the protected ref is not bound — the local trunk ref is immutable (`upstream_integration.rs:159`) | Same accepted-leak class as R1; the governed `but` merge is gated, the bypass paths are named, and the steel-trap pre-receive (push) + forge-side rules (forge) are the deferred closures. The build must NOT claim the merge gate binds forge/push merges. |
| R12 | **Human-as-fleet-owner trust (v1)** | Accepted | The desktop user is a trusted superuser over the agent fleet — manages governance config by implicit ownership, not a checked grant; anyone at the keyboard, or a compromised desktop session, can manage the fleet (grant/revoke agent permissions, edit gates, manage groups) | Personal-tenant trust — the human owns the repo + its agents; the orchestrator is trusted the same way ("your machine, your keys"). **Agent authorization is unaffected** (agents stay functionally bound, no superuser path). Named as a v1 assumption (`02-roles.md`). The build must NOT present the human principal as a checked/authenticated identity. Future closure: a real per-human authenticated principal (key-mint / a checked `administration:write` entry in `permissions.toml`). |
| R13 | **Agent weakens governance by editing config files directly** | Accepted (out of scope) | An agent with `contents:write` can `fs.write` `.gitbutler/{permissions,gates}.toml` and commit it; the ref-pin only blocks *same-change* self-escalation, so a config-weakening change that lands on the target ref takes effect for future actions | **Strict scope line — GitButler governs its own action surface**: the `but` actions + the `but perm`/`but group`/gate-edit config path (gated by `administration:write`, which the agent role presets exclude by default). A direct `fs.write`/raw-git edit to the config files is **outside governance scope** — the same accepted-leak class as the fence (R1, Assumption 1). We deliberately do **not** add client/harness commit hooks (that would govern the external harness, which this system does not do) and do **not** fast-follow a stronger config-path enforcement. Irrigation applies: the `but` path is the cheap path; a raw config edit is still inert until committed and landed via the governed merge, and config-touching changes are reviewable as a diff. The build must NOT present direct-file edits as prevented. |
| R14 | **N-API / direct-caller bypass of the `but-api` enforcement seam** | Medium (Accepted residual) | The enforcement seam is only effective if **all four callers** route through `but-api`'s gated wrappers. `but-napi` (the N-API bindings driving the Electron *lite* app) may call `but-workspace`/`but-core` directly, skipping `but-api`'s `_with_authz` wrapper / pre-call guard — an ungoverned lower-level call would land outside the gates, the same class of hole as a missed route (R8) but reached from a *different binding* | **Audit the N-API surface; route every consequential N-API entry point through `but-api`** (or its gated action). A build-gate grep-audit asserts each consequential `but-napi` route is `_with_authz`-wrapped / pre-call-guarded — no ungoverned lower-level call (T-AUTHZ-016b, mapping the UC-AUTHZ-02 N-API AC). A call that escapes the audit is the **same accepted-leak class as R1** (residual **Accepted**) for the POC; the steel-trap boundary is the eventual closure. The build must NOT claim the seam is enforced for N-API callers it has not audited. |

## The risks the reviewer must treat as honesty tests
- **R1 must stay named, never quietly "mitigated" into looking closed.** A build report that implies the fence is a security boundary is a misrepresentation. The fence is leaky; the value is the irrigation gradient.
- **R6's review-store forgeability must stay named.** The merge gate's review requirement is only as strong as the assumption that reviews arrive through the governed `but` action; a direct DB write forges an approval (same accepted-leak class as R1). Presenting the local review store as tamper-proof is the same misrepresentation class as R1. (R6 is **High** for exactly this reason; the HMAC/Ed25519 hardening is the deferred closure.)
- **R11 and R2's identity residual must stay named too.** The merge gate binds the *governed* action, not forge/push merges; identity confinement is partial (sub-process env re-export is not closed). Presenting either as complete is the same misrepresentation class as R1.
- **R14's N-API residual must stay named.** The enforcement seam is effective only for callers that route through `but-api`; an unaudited N-API path that calls a lower-level crate directly is an accepted residual, not a closed boundary. Presenting the seam as enforced for *every* binding without the N-API audit is the same misrepresentation class as R1.
- **R12 and R13 are accepted v1 properties, not closed boundaries.** The human fleet-owner is *trusted* (not authenticated); a direct config-file edit by an agent is *out of governance scope* (we govern `but`, not the filesystem/harness — same leak class as R1). Presenting either as enforced would be the same misrepresentation class as R1.
- **R5/R3 are grep-asserted invariants.** If either fails (a role name in enforcement, or the `Permission` lock overloaded), the slice is not done regardless of green tests.

## Criteria-tally note (A3.3)
The e2e testing criteria are tallied **by test type** (integration / component / api-contract / build-gate / e2e-automated), **not by risk severity** — so R6's Medium→High upgrade and the new R14 add **no ripple** to the criteria-type counts in `11-e2e-testing-criteria.md`. The only criterion-count change from the risk edits is the **+1 build-gate** criterion (T-AUTHZ-016b) that operationalizes R14's N-API audit.
