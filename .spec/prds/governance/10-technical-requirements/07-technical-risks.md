# Technical Risks

Risks carried into implementation, with the mitigation each gets in this slice. The first two are **accepted by design** (the POC's scope line); the rest are correctness risks the build must close.

**Risk register: 23 risks.** R6 is **High** (the merge gate's review store is forgeable by direct DB write); R14 (**Medium**, Accepted residual) covers the N-API / direct-caller bypass of the `but-api` enforcement seam; R18 (**High**, R6 class) covers the local-review loop-sourced-receipt forgeability. R15–R17 (STEER, v1.4.0) and R18–R23 (LPR, v1.5.0) are folded in from their enrichments — full statements in [`enrichments/v1.4.0-capability-aware-denials/05-delta-replan.md`](../../enrichments/v1.4.0-capability-aware-denials/05-delta-replan.md) §3 and [`enrichments/v1.5.0-local-agent-pr/05-delta-replan.md`](../../enrichments/v1.5.0-local-agent-pr/05-delta-replan.md) §3.

## The risks the reviewer must treat as honesty tests

- **R1 must stay named, never quietly "mitigated" into looking closed.** A build report that implies the fence is a security boundary is a misrepresentation. The fence is leaky; the value is the irrigation gradient.
- **R6's review-store forgeability must stay named.** The merge gate's review requirement is only as strong as the assumption that reviews arrive through the governed `but` action; a direct DB write forges an approval (same accepted-leak class as R1). Presenting the local review store as tamper-proof is the same misrepresentation class as R1. (R6 is **High** for exactly this reason; the HMAC/Ed25519 hardening is the deferred closure.)
- **R11 and R2's identity residual must stay named too.** The merge gate binds the _governed_ action, not forge/push merges; identity confinement is partial (sub-process env re-export is not closed). Presenting either as complete is the same misrepresentation class as R1.
- **R14's N-API residual must stay named.** The enforcement seam is effective only for callers that route through `but-api`; an unaudited N-API path that calls a lower-level crate directly is an accepted residual, not a closed boundary. Presenting the seam as enforced for _every_ binding without the N-API audit is the same misrepresentation class as R1.
- **R12 and R13 are accepted v1 properties, not closed boundaries.** The human fleet-owner is _trusted_ (not authenticated); a direct config-file edit by an agent is _out of governance scope_ (we govern `but`, not the filesystem/harness — same leak class as R1). Presenting either as enforced would be the same misrepresentation class as R1.
- **R15–R23 (STEER + LPR enrichment risks) must stay named, never quietly "mitigated" into looking closed.** R18's loop-sourced-receipt forgeability (High, R6 class), R19's tag spoofability via handle re-export, R21's keep-reviews-local preference-not-boundary flip, and R23's tag-control-path DB forgery are the same misrepresentation class as R1/R6 — presenting any as a hardened boundary is dishonest. R20's comment-body injection is an accepted L2/harness residual. R22's drive-layer distinct-from-author/resolver-identity constraints (T-LPR-043/044) are real integrity checks but narrow _cross-principal_ forgery only — a single-principal drive trail is NOT multi-party review. R15's config-derived-string injection into `message`/`unmet[]` is NOT covered by the STEER closed-catalog grep (that covers only the new fields). R16/R17 are Low and mitigated-by-design/accepted.
- **R5/R3 are grep-asserted invariants.** If either fails (a role name in enforcement, or the `Permission` lock overloaded), the slice is not done regardless of green tests.

## Criteria-tally note (A3.3)

The e2e testing criteria are tallied **by test type** (integration / component / api-contract / build-gate / human-gate / e2e-automated), **not by risk severity** — so R6's Medium→High upgrade, R14, and the enrichment risks R15–R23 add **no ripple** to the criteria-type counts beyond the criteria the enrichments themselves carry. The enrichment criteria (T-STEER-001..031, T-LPR-001..044 + T-LPR-029h) are folded into `11-e2e-testing-criteria.md` (I5), taking the totals to **205 criteria** (118 integration · 38 component · 19 api-contract · 25 build-gate · 1 human-gate · 4 e2e-automated). Of the enrichment risks, only R22 adds _test_ rows (T-LPR-043/044, both integration-test); the rest add no ripple to the type counts.
