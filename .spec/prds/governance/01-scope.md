# Scope

**Scope Posture**: Full feature (default — a complete, fully-polished POC slice).

This PRD is the **permissions + two-gates POC** that turns GitButler into a policy-enforcement layer for orchestrated agents. The boundary sentence for every item: **this initiative enforces GitButler git actions through a functional permission set and two deterministic gates — it does not block git, and it does not drive the agent's reasoning.** Everything opinionated lives in committed config; the engine ships mechanism only. It is native to GitButler and grounded in GitButler's own crates.

## Two load-bearing assumptions (constraints, not features)

These are the explicit edges of the POC, stated so no reader mistakes them for gaps to be "fixed" inside this slice.

1. **The system does NOT block git.** Forcing an agent through the `but` CLI (so its commits/merges pass through GitButler's gated actions) is delegated to **git commit hooks or harness hooks** — an *accepted-leak* fence. `--no-verify`, an agent editing `.git/hooks` / `core.hooksPath`, or raw git plumbing can bypass it. That residual is **knowingly accepted for the POC** (see [Known Limitations](#known-limitations)); per the irrigation thesis, the governed path only has to be *cheaper than* the bypass, not impossible. The *steel-trap* boundary (a server-side pre-receive on a bare repo, or an OS sandbox where GitButler is the agent's sole channel to the repo) is **future work, explicitly out of this slice.**

2. **The system does NOT drive agent logic.** The harness/orchestrator runs the agents and owns their reasoning, prompts, and tool loop. GitButler is used **purely as a commit/orchestration policy-enforcement system** — it governs git *actions*, never the agent's *thinking*. There is no agent runtime, no turn loop, no system-prompt assembly in this PRD (those are agent-runtime concerns, deliberately out of scope).

## In Scope

### Functional permission system — AUTHZ
- A functional, GitHub-mirrored permission catalog (`metadata:read`, `contents:read/write`, `pull_requests:read/write`, `reviews:write`, `comments:write`, `merge`, `statuses:read/write`, `administration:read/write`); the **enforced subset** (each with a real gated GitButler route) is `{contents:read/write, pull_requests:write, reviews:write, comments:write, merge, administration:write}`, the rest parseable-but-catalog-only this slice. Stored per-principal in committed, ref-pinned `.gitbutler/permissions.toml`
- A new `but-authz` crate owning `Authority` / `AuthoritySet` / `Principal` / `Group`, `authorize(principal, action)`, the deny contract, and the ref-pinned config loader — named distinctly from GitButler's existing `Permission` (a repo-access lock), which it must not overload
- Named-role presets (read/triage/write/maintain/admin) that **desugar to functional sets at load**; superuser = all permissions; a raw functional list loads without a role
- Per-action enforcement at the GitButler action boundary (`but-api` — a `_with_perm`-style acquisition for local lock-taking actions, a pre-call `authorize()` guard for async forge actions), returning the structured agent-readable denial `{code, message, remediation_hint}` + exit 1; **no enforcement path references a role name**. The seam is effective only for callers that route through `but-api` (Tauri / `but` CLI / TUI / N-API) — the N-API surface must be audited (R14)
- **Fail-closed by default**: an unknown principal, a missing `BUT_AGENT_HANDLE`, or an unreadable/malformed governed config at the target ref **denies** the action — never defaults to allow
- **Identity confinement**: a dispatched agent's `but` calls are pinned to its own principal via an injected `BUT_AGENT_HANDLE`; a `--as <other>` style call is denied while dispatched (the residual — a process re-setting its own env — is an accepted leak, see Known Limitations)
- **Config-change authority**: `administration:write` gates `but perm`/`but group` and any change touching `.gitbutler/{permissions,gates}.toml`

### Principal grouping — GRPS
- Grant functional permissions to **users or groups**; a principal's effective `AuthoritySet` is the **union** of its own grants and the grants of every group it belongs to (GitHub teams model)
- Group definitions, grants, and membership are committed, ref-pinned governed config, read at the **target ref** when authorizing — so a change that adds its author to a privileged group is **ineffective** until committed to the target branch (no self-escalation via the working tree or a feature head)
- `but group` CLI to create groups, grant functional permissions to a group, manage membership, and **delete a group** (each itself gated by `administration:write`)

### The two gates (control plane) — GATES
- **Branching-mechanism-agnostic**: the gates apply to the git *action* (commit, integrate/merge, push) **whichever branching mechanism produced it** — virtual branches/stacks (the default, one shared working tree), normal-git single-branch, or **opt-in worktrees** (`but-worktrees`). Worktrees are never forced; when used, their commit/integrate/push are gated identically. The build enumerates every commit/ref-advancing entry point (`commit_engine`, branch apply, `but-worktrees::integrate`, the forge PR-merge, push) and gates each
- **Commit gate** (at GitButler's commit narrow-waist, principal resolved from `BUT_AGENT_HANDLE`): the principal must hold `contents:write`; a protected target branch rejects a direct commit with `{code:"branch.protected", branch, …}` — landing on a protected branch must go through review → merge
- **Merge gate** (at GitButler's PR-merge action): the principal must hold `merge`, **and** the configured review requirement must be satisfied — ≥1 approving review, distinct from the author, from each required group, valid at the **current head**; an unmet requirement returns `{code:"gate.review_required", unmet:[…]}`. The gate binds the *governed* `but` merge action; a merge performed entirely on the forge (auto-merge / UI) or by raw push is an accepted-leak bypass (steel-trap closes it). The review requirement reads the local `local_review_verdicts` store, which is **forgeable by direct DB write (R6, High)** — sound only for reviews submitted through the governed `but review` action
- The gate engine is **unopinionated**: thresholds (`min_approvals`, `require_distinct_from_author`, `require_approval_from_group`) and which branches are protected live entirely in committed, ref-pinned `.gitbutler/gates.toml`; the engine bakes in no roles and no workflow opinion

### Governed orchestration loop — LOOP
- An end-to-end demonstration: an implementer principal (`contents:write`, no `merge`), a reviewer principal (`reviews:write`), and a human maintainer principal (`merge` + `administration:write`) run implement→review→merge through GitButler's gated actions
- **Role separation emerges from the functional permission set, not from code**: the implementer is denied `merge` legibly; the reviewer's edits are inert because it lacks `contents:write`; only the maintainer can land, and only after the configured reviews exist at head
- **The governed path is traversable** (the irrigation half): a denied agent's `remediation_hint` names a governed next action that, when followed, succeeds — so the demo proves the channel is cheap, not only that the bypass is blocked
- **Human-at-feature-level + AI-at-code-level expressed purely as config**: a `code-reviewers` group (AI agents, `reviews:write`) and a `maintainers` group (humans, `merge`), with `gates.toml` requiring an approval from *each* — so the AI catches code-level misses (e.g. a wholly-missing feature) and the human owns quality at the feature level, with zero role-specific enforcement code
- The loop's "open a PR" / "submit a review" steps gate on the governed `but pr` / `but review` actions (`pull_requests:write` / `reviews:write` / `comments:write`), each permission-checked at the `but-api` boundary; the demo assumes **honest review submission** (a forged review via direct DB write is out of scope, R6)

### Governance management UI — MGMT
- A human **governance-management surface in the SvelteKit/Tauri desktop app** (`apps/desktop`): a new **admin-only Project Settings page** (`projectSettingsPages.ts` `adminOnly:true`, rendered via `SettingsModalLayout`) with four tabs — Principals · Groups · Branch Gates · Rules
- Per-agent management of functional permissions (role preset + functional toggles, inherited rows read-only, **batch-saved**), group membership (incl. group delete + destructive-action confirmations), branch gates (`gates.toml`), and per-agent automation rules (reusing the existing `rules/` components via a new optional `principalId` prop)
- A **governed front-end**: every write calls the same `but-api` function the CLI uses, exposed via a Tauri command → generated `packages/but-sdk`; changes are **pending until committed** to the governance ref (with defined commit semantics); the surface is **read-only** without `administration:write`; self-escalation is not optimistically applied — the UI is never a bypass
- **No new design-system work**: reuses `packages/ui` + `apps/desktop/src/components/shared` + the existing `settings/` and `rules/` surfaces (~6 thin new components + 1 `RulesList` prop + 1 settings-page entry). A **desktop CT/Vitest config** is a build prerequisite for the MGMT component tests (B14 / T-MGMT-000). Specialists: `sveltekit-*` + `tauri-*` + `frontend-designer`

## Out of Scope

### Deferred to later governance layers (named, not denied)
- **Ed25519-signed reviews & comments** — in this POC, reviews/approvals are attributed by principal identity; cryptographic signing of the review artifact is the next hardening pass. The intermediate hardening is an **HMAC integrity check** on `local_review_verdicts` keyed by a repo-local admin secret (C3), closing R6's direct-DB-write forgeability
- **A full multi-clause gate** — this slice ships a *minimal* review requirement, not a full clause set (task-requirement, conflict-free, signed-commits, assigned-reviewer, author-is-assignee)
- **Auto-run validation feeding the gate** (`on: pull_request` auto-trigger + checks@head) — required-checks-green is deferred; the POC's gate is review + permission only
- **Config supremacy / break-glass** (an audited override; grep-asserted absence of `force`/`skip_gate`) — deferred; the POC has no override path at all
- **The steel-trap transport boundary** — server-side pre-receive on a bare repo, or an OS sandbox where GitButler is the agent's sole repo channel; the physical closure of Assumption 1's accepted leak, explicitly future work
- **First-admin onboarding UI + `but governance init` build** — the full onboarding workflow and a new `but governance init` verb are a **fast-follow** (C1); in v1 the trusted human fleet-owner (R12) creates `.gitbutler/permissions.toml` directly, and fail-closed-on-missing-config is already covered (UC-AUTHZ-04)
- **A `but perm set` (overwrite) CLI verb** — the MGMT batch-save is implemented as a sequence of the existing `but perm grant`/`revoke` verbs; a new overwrite verb is deferred (C2)

### Genuinely out (non-goals)
- **Blocking git / `fs.write` governance** — governed GitButler actions suffice for the POC (a reviewer's edits are inert because it cannot land them); raw git is fenced only by the accepted-leak hook
- **Driving the agent's reasoning** — no agent runtime, turn loop, methodology prompt, or raw-git ban enforced by GitButler; the harness owns the agent (Assumption 2)
- **A new *agent* UI, or a new app** — the agent-facing *enforcement* stays CLI/action-boundary (no agent UI), and MGMT creates **no new app**: it is a **feature inside the existing SvelteKit desktop app** (`apps/desktop`), a settings-section *state*, not a new top-level route and not a new application
- **The non-git GitHub permission surface** (Actions, Secrets, Environments, Deployments, Webhooks, Pages, Dependabot, …) — the POC scopes to the git + governance core
- **User-defined named permission templates** — beyond the built-in read/triage/write/maintain/admin presets, custom user-named bundles are deferred
- **Real-time session-permission refresh** — a permission change takes effect once committed to the target ref; live in-session refresh is deferred

## Known Limitations

Explicitly acknowledged gaps in the POC — known to the design, documented for reviewers, tracked for follow-up:

| Limitation | Impact | Mitigation in POC | Follow-up |
|---|---|---|---|
| The fence is an **accepted leak** (client git hooks / harness hooks) | An agent with a raw shell, `--no-verify`, `core.hooksPath` edits, or git plumbing can commit/merge outside GitButler, bypassing the gates | Per the irrigation thesis, the governed path is the cheapest path; honest agents route through `but`; the leak is named, not hidden | Steel-trap boundary: server-side pre-receive on a bare repo, or an OS sandbox |
| The merge gate binds the **governed `but` merge action only** | A merge performed entirely on the forge (auto-merge / UI / CI) or by raw `git push` to the protected ref is not bound (GitButler's trunk ref is immutable locally — landing is a remote-forge op) | The governed merge is gated; the bypass paths are the same accepted-leak class as the fence | The steel-trap pre-receive closes the push path; forge-side rules close the forge path |
| The **enforcement seam binds only callers that route through `but-api`** (R14) | `but-napi` (N-API / Electron lite) may call `but-workspace`/`but-core` directly, skipping `but-api`'s `_with_authz` wrapper — an ungoverned bypass for any unaudited N-API route | Audit every N-API entry point (build-gate grep-assert, T-AUTHZ-016b); route each consequential route through `but-api`. An unaudited route is the same accepted-leak class as the fence (R1) | The steel-trap boundary is the eventual closure for the whole accepted-leak class |
| Identity rests on an **injected `BUT_AGENT_HANDLE`** | `--as` is denied, but a process that can set its own env can re-export `BUT_AGENT_HANDLE` and present another handle | Personal-tenant trusts the orchestrator; `--as` denial covers the in-band case | Per-agent key-mint authorization (each agent proves possession of an agent-scoped secret) |
| An `administration:write` holder can **bootstrap config escalation** | Such a principal could rewrite `gates.toml` to unprotect a branch, then escalate `permissions.toml` on it | The ref-pin stops *feature-head* self-escalation; config-change authority is itself gated. Same-ref bootstrap by an admin who can also reach the fence is an accepted residual | A bootstrap invariant (un-protecting a branch must itself clear the existing gate) is a hardening follow-up |
| The local **review record is forgeable / not integrity-protected** (R6, **High**) | An agent with DB/filesystem write access can INSERT a forged approving review directly into `local_review_verdicts` (`principal_id`/`verdict`/`head_oid`); the merge gate is then **trivially satisfiable by direct DB write** — the review requirement is only as strong as the assumption that reviews arrive through the governed `but` action | Accepted for the POC — the **same accepted-leak class as the fence (R1)**. The merge gate is sound only for reviews submitted through the governed `but review` action; the LOOP demo assumes honest review submission; a direct DB write is named, not hidden, and the gate tests exercise the governed path, not the forgeable direct write | An **HMAC integrity check** keyed by a repo-local admin secret (C3), then Ed25519-signed review artifacts stored tamper-evidently |
| Reviews are **attributed, not cryptographically signed** | A forged review artifact is not detectable by signature in this slice | Principal identity carries attribution; the gate requires a *distinct* principal | Ed25519 review signing |
| The merge gate is a **minimal review requirement**, not a full clause set | A merge can pass the POC gate while a richer policy (conflict-free, checks-green, task-requirements) would block | Documented; the clause set is config-extensible by design | A full clause set + auto-validation checks |
| A permission/group change is **inert until committed to the target ref** | `but perm grant` / `but group` edits the file; the grant does nothing until committed and the target ref advances | Documented; the ref-pin is the security property (no self-escalation), not a bug | Real-time session-permission refresh |
| **No override path** exists in the POC | A genuinely-stuck merge has no break-glass | Acceptable for a POC; the only lever is a governed `gates.toml` change | A config-gated, audited override |
| **First-admin bootstrap is by trusted ownership, not a built workflow** | On a project with no `.gitbutler/permissions.toml`, governance fails closed (UC-AUTHZ-04); there is no in-app onboarding UI to create the first admin in the POC | The bootstrap path is the trusted human fleet-owner (R12) creating the config file directly (they own the repo) — the chicken-and-egg dissolves under personal-tenant trust; correctness (fail-closed on empty/missing config) is already covered by UC-AUTHZ-04 / T-AUTHZ-029 | A `but governance init` convenience verb + a full onboarding UI workflow (C1) — named fast-follow, not built in the POC |
| A **human is a trusted superuser over the agent fleet** (v1) | The desktop user (the fleet owner) manages governance config by implicit ownership, not a checked grant; anyone at the keyboard, or a compromised desktop session, can manage the fleet (grant/revoke agent permissions, edit gates) | Personal-tenant trust — the human owns the repo + its agents (same trust class as the orchestrator); **agent authorization is unaffected** (agents stay functionally bound, no superuser path). Named, not hidden (`02-roles.md`) | A real per-human authenticated principal (key-mint / checked `administration:write`) so the model scales beyond one trusted human |
| An **agent can weaken/disable governance by editing the config files directly** | An agent with `contents:write` can `fs.write` `.gitbutler/{permissions,gates}.toml` and commit it (`fs.write` is ungoverned — Assumption 1); the ref-pin only blocks *same-change* self-escalation, so a config-weakening change that lands on the target ref takes effect for future actions | **Out of governance scope by design** — GitButler governs its own `but` actions + the `but` config-management path (gated by `administration:write`); it does **not** govern the filesystem or external harnesses (same accepted-leak class as the fence, R1). The `but` path is gated; a raw-file edit is uphill and still inert until committed + landed via the governed merge, and config-touching changes are reviewable as a diff. Users who want to harden this can protect the config files with a commit/harness hook that rejects writes to `.gitbutler/*.toml` — a user-side option, outside this PRD's scope. Named, not hidden | None (strict scope line, not a fast-follow) — the steel-trap boundary of R1 is the eventual closure for the whole accepted-leak class |
