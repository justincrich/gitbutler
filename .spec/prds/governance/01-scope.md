# Scope

**Scope Posture**: Full feature (default â€” a complete, fully-polished POC slice).

This PRD is the **permissions + two-gates POC** that turns GitButler into a policy-enforcement layer for orchestrated agents. The boundary sentence for every item: **this initiative enforces GitButler git actions through a functional permission set and two deterministic gates â€” it does not block git, and it does not drive the agent's reasoning.** Everything opinionated lives in committed config; the engine ships mechanism only. It is native to GitButler and grounded in GitButler's own crates.

## Two load-bearing assumptions (constraints, not features)

These are the explicit edges of the POC, stated so no reader mistakes them for gaps to be "fixed" inside this slice.

1. **The system does NOT block git.** Forcing an agent through the `but` CLI (so its commits/merges pass through GitButler's gated actions) is delegated to **git commit hooks or harness hooks** â€” an _accepted-leak_ fence. `--no-verify`, an agent editing `.git/hooks` / `core.hooksPath`, or raw git plumbing can bypass it. That residual is **knowingly accepted for the POC** (see [Known Limitations](#known-limitations)); per the irrigation thesis, the governed path only has to be _cheaper than_ the bypass, not impossible. The _steel-trap_ boundary (a server-side pre-receive on a bare repo, or an OS sandbox where GitButler is the agent's sole channel to the repo) is **future work, explicitly out of this slice.**

2. **The system does NOT drive agent logic.** The harness/orchestrator runs the agents and owns their reasoning, prompts, and tool loop. GitButler is used **purely as a commit/orchestration policy-enforcement system** â€” it governs git _actions_, never the agent's _thinking_. There is no agent runtime, no turn loop, no system-prompt assembly in this PRD (those are agent-runtime concerns, deliberately out of scope).

## In Scope

### Functional permission system â€” AUTHZ

- A functional, GitHub-mirrored permission catalog (`metadata:read`, `contents:read/write`, `pull_requests:read/write`, `reviews:write`, `comments:write`, `merge`, `statuses:read/write`, `administration:read/write`); the **enforced subset** (each with a real gated GitButler route) is `{contents:read/write, pull_requests:write, reviews:write, comments:write, merge, administration:write}`, the rest parseable-but-catalog-only this slice. Stored per-principal in committed, ref-pinned `.gitbutler/permissions.toml`
- A new `but-authz` crate owning `Authority` / `AuthoritySet` / `Principal` / `Group`, `authorize(principal, action)`, the deny contract, and the ref-pinned config loader â€” named distinctly from GitButler's existing `Permission` (a repo-access lock), which it must not overload
- Named-role presets (read/triage/write/maintain/admin) that **desugar to functional sets at load**; superuser = all permissions; a raw functional list loads without a role
- Per-action enforcement at the GitButler action boundary (`but-api` â€” a `_with_perm`-style acquisition for local lock-taking actions, a pre-call `authorize()` guard for async forge actions), returning the structured agent-readable denial `{code, message, remediation_hint}` + exit 1; **no enforcement path references a role name**. The seam is effective only for callers that route through `but-api` (Tauri / `but` CLI / TUI / N-API) â€” the N-API surface must be audited (R14)
- **Fail-closed by default**: an unknown principal, a missing `BUT_AGENT_HANDLE`, or an unreadable/malformed governed config at the target ref **denies** the action â€” never defaults to allow
- **Identity confinement**: a dispatched agent's `but` calls are pinned to its own principal via an injected `BUT_AGENT_HANDLE`; a `--as <other>` style call is denied while dispatched (the residual â€” a process re-setting its own env â€” is an accepted leak, see Known Limitations)
- **Config-change authority**: `administration:write` gates `but perm`/`but group` and any change touching `.gitbutler/{permissions,gates}.toml`

### Principal grouping â€” GRPS

- Grant functional permissions to **users or groups**; a principal's effective `AuthoritySet` is the **union** of its own grants and the grants of every group it belongs to (GitHub teams model)
- Group definitions, grants, and membership are committed, ref-pinned governed config, read at the **target ref** when authorizing â€” so a change that adds its author to a privileged group is **ineffective** until committed to the target branch (no self-escalation via the working tree or a feature head)
- `but group` CLI to create groups, grant functional permissions to a group, manage membership, and **delete a group** (each itself gated by `administration:write`)

### The two gates (control plane) â€” GATES

- **Branching-mechanism-agnostic**: the gates apply to the git _action_ (commit, integrate/merge, push) **whichever branching mechanism produced it** â€” virtual branches/stacks (the default, one shared working tree), normal-git single-branch, or **opt-in worktrees** (`but-worktrees`). Worktrees are never forced; when used, their commit/integrate/push are gated identically. The build enumerates every commit/ref-advancing entry point (`commit_engine`, branch apply, `but-worktrees::integrate`, the forge PR-merge, push) and gates each
- **Commit gate** (at GitButler's commit narrow-waist, principal resolved from `BUT_AGENT_HANDLE`): the principal must hold `contents:write`; a protected target branch rejects a direct commit with `{code:"branch.protected", branch, â€¦}` â€” landing on a protected branch must go through review â†’ merge
- **Merge gate** (at GitButler's PR-merge action): the principal must hold `merge`, **and** the configured review requirement must be satisfied â€” â‰¥1 approving review, distinct from the author, from each required group, valid at the **current head**; an unmet requirement returns `{code:"gate.review_required", unmet:[â€¦]}`. The gate binds the _governed_ `but` merge action; a merge performed entirely on the forge (auto-merge / UI) or by raw push is an accepted-leak bypass (steel-trap closes it). The review requirement reads the local `local_review_verdicts` store, which is **forgeable by direct DB write (R6, High)** â€” sound only for reviews submitted through the governed `but review` action
- The gate engine is **unopinionated**: thresholds (`min_approvals`, `require_distinct_from_author`, `require_approval_from_group`) and which branches are protected live entirely in committed, ref-pinned `.gitbutler/gates.toml`; the engine bakes in no roles and no workflow opinion

### Governed orchestration loop â€” LOOP

- An end-to-end demonstration: an implementer principal (`contents:write`, no `merge`), a reviewer principal (`reviews:write`), and a human maintainer principal (`merge` + `administration:write`) run implementâ†’reviewâ†’merge through GitButler's gated actions
- **Role separation emerges from the functional permission set, not from code**: the implementer is denied `merge` legibly; the reviewer's edits are inert because it lacks `contents:write`; only the maintainer can land, and only after the configured reviews exist at head
- **The governed path is traversable** (the irrigation half): a denied agent's `remediation_hint` names a governed next action that, when followed, succeeds â€” so the demo proves the channel is cheap, not only that the bypass is blocked
- **Human-at-feature-level + AI-at-code-level expressed purely as config**: a `code-reviewers` group (AI agents, `reviews:write`) and a `maintainers` group (humans, `merge`), with `gates.toml` requiring an approval from _each_ â€” so the AI catches code-level misses (e.g. a wholly-missing feature) and the human owns quality at the feature level, with zero role-specific enforcement code
- The loop's "open a PR" / "submit a review" steps gate on the governed `but pr` / `but review` actions (`pull_requests:write` / `reviews:write` / `comments:write`), each permission-checked at the `but-api` boundary; the demo assumes **honest review submission** (a forged review via direct DB write is out of scope, R6)

### Governance management UI â€” MGMT

- A human **governance-management surface in the SvelteKit/Tauri desktop app** (`apps/desktop`): a new **admin-only Project Settings page** (`projectSettingsPages.ts` `adminOnly:true`, rendered via `SettingsModalLayout`) with four tabs â€” Principals Â· Groups Â· Branch Gates Â· Rules
- Per-agent management of functional permissions (role preset + functional toggles, inherited rows read-only, **batch-saved**), group membership (incl. group delete + destructive-action confirmations), branch gates (`gates.toml`), and per-agent automation rules (reusing the existing `rules/` components via a new optional `principalId` prop)
- A **governed front-end**: every write calls the same `but-api` function the CLI uses, exposed via a Tauri command â†’ generated `packages/but-sdk`; changes are **pending until committed** to the governance ref (with defined commit semantics); the surface is **read-only** without `administration:write`; self-escalation is not optimistically applied â€” the UI is never a bypass
- **No new design-system work**: reuses `packages/ui` + `apps/desktop/src/components/shared` + the existing `settings/` and `rules/` surfaces (~6 thin new components + 1 `RulesList` prop + 1 settings-page entry). A **desktop CT/Vitest config** is a build prerequisite for the MGMT component tests (B14 / T-MGMT-000). Specialists: `sveltekit-*` + `tauri-*` + `frontend-designer`

### Capability-aware denials â€” STEER (v1.4.0 enrichment, folded)

- **A standard capability-aware denial shape.** Three additive fields on the existing `Denial` contract â€” `class`, `held_permissions`, `authorized_actions` â€” plus a `do_not` prohibition, carried uniformly by every actor-correctable denial across both gates and the authz primitive. `code`, `message`, `remediation_hint`, and `unmet` are **preserved unchanged** (back-compat + human readability)
- **A derived, intent-scoped `authorized_actions` menu.** Computed deterministically as `effective_set âˆ© routeâ†’Authority table`, filtered to the actions relevant to the denied intent, each entry `{command, effect}`. The single source of truth is the same routeâ†’Authority table the gates enforce against (no lying menu)
- **A recoverability `class`** (`actor_correctable` | `operator_required`) that tells the agent whether to adapt-now or stop-and-escalate, with graceful degradation to the existing vertical "hand off / ask an admin" path when no lateral action exists
- **Self-discovery on demand.** Team/group membership and the full permission/action set are reachable via `but perm list` (self-scoped â€” a **Sprint 05 deliverable this enrichment depends on**) surfaced as one of the `authorized_actions`, optionally via a net-new `but whoami` / `but can-i` entry point STEER builds. Provenance is served, not pushed inline
- **Reference agent-priming guidance (L2, non-enforced).** A short "denials are redirects; affordances are not orders; never bypass" primer a harness/orchestrator MAY adopt â€” documented as reference, **not** an enforcement requirement (Stance 6)
- **Traversability + anti-injection invariants (L3).** Every action a denial offers must actually succeed for that caller (no "lying menu"); steering text is drawn from a closed, code-owned catalog (no free-form or injectable content). Proven by extending the existing `governed_loop` test harness

### Local Agent PR / governed-review parity â€” LPR (v1.5.0 enrichment, folded)

- **A local PR object (Full).** Three new **additive** `but-db` tables (`local_review_assignments`, `local_review_comments`, `local_review_meta`) on the existing `SchemaVersion`-style migration system. The PR lifecycle state is **derived** (computed from commits + verdict-at-head + open assignments), not stored as a fourth truth
- **A `but review` command/API surface.** Request/assign a reviewer, post/list/resolve comments, and query review status â€” built on the **already-shipped** authorities (`PullRequestsWrite` / `ReviewsWrite` / `CommentsWrite`); **no new authority is introduced**. Drive-layer integrity lives at the `but-api` boundary (assign distinct-from-author; resolve resolver-identity). Like the existing `approve_review`, these are local-cache writes (no network, no `DryRun` guard)
- **A per-project "keep PRs local" setting.** A per-project operator preference on `gitbutler_project::Project` (same class as `forge_override`) that makes agent-authored reviews default to the local layer â€” **NOT** `administration:write`-gated, **NOT** ref-pinned config (R12 trusted-desktop). Remote mirroring is gated behind this setting (the bridge itself is named-for-later)
- **An automatic agent-PR tag.** A review/PR opened by a principal whose committed `.gitbutler/permissions.toml` entry **declares `kind = "agent"`** is auto-tagged `agent-authored` (a label, `ForgeReview.labels` precedent) â€” the distinction is a **declared config fact** (additive optional `kind` field on `PrincipalWire`, read at the target ref), NOT inferred from handle-resolution. Cached in a dedicated `local_review_meta` row, never a comment body
- **A `but-*` skill contract.** The `but-*` skills auto-set these PRs local when used (the skill workflow invokes the local `but review` surface with "keep PRs local"); the skills assume the local surface exists and is the default
- **A reconciler usage model + the auto-hook.** The orchestrator-as-reconciler pattern (assignment â†’ dispatch reviewer; unresolved comment â†’ remediation; approved-verdict-at-head â†’ merge). The existing `but-rules` (Trigger â†’ Filter â†’ Action) engine hosts an auto "review-requested" hook on commit (extending the shipped engine with a new commit `Trigger` + review-assignment `Action` variant)

## Out of Scope

### Deferred to later governance layers (named, not denied)

- **Ed25519-signed reviews & comments** â€” in this POC, reviews/approvals are attributed by principal identity; cryptographic signing of the review artifact is the next hardening pass. The intermediate hardening is an **HMAC integrity check** on `local_review_verdicts` keyed by a repo-local admin secret (C3), closing R6's direct-DB-write forgeability
- **A full multi-clause gate** â€” this slice ships a _minimal_ review requirement, not a full clause set (task-requirement, conflict-free, signed-commits, assigned-reviewer, author-is-assignee)
- **Auto-run validation feeding the gate** (`on: pull_request` auto-trigger + checks@head) â€” required-checks-green is deferred; the POC's gate is review + permission only
- **Config supremacy / break-glass** (an audited override; grep-asserted absence of `force`/`skip_gate`) â€” deferred; the POC has no override path at all
- **The steel-trap transport boundary** â€” server-side pre-receive on a bare repo, or an OS sandbox where GitButler is the agent's sole repo channel; the physical closure of Assumption 1's accepted leak, explicitly future work
- **First-admin onboarding UI + `but governance init` build** â€” the full onboarding workflow and a new `but governance init` verb are a **fast-follow** (C1); in v1 the trusted human fleet-owner (R12) creates `.gitbutler/permissions.toml` directly, and fail-closed-on-missing-config is already covered (UC-AUTHZ-04)
- **A `but perm set` (overwrite) CLI verb** â€” the MGMT batch-save is implemented as a sequence of the existing `but perm grant`/`revoke` verbs; a new overwrite verb is deferred (C2)

### Genuinely out (non-goals)

- **Blocking git / `fs.write` governance** â€” governed GitButler actions suffice for the POC (a reviewer's edits are inert because it cannot land them); raw git is fenced only by the accepted-leak hook
- **Driving the agent's reasoning** â€” no agent runtime, turn loop, methodology prompt, or raw-git ban enforced by GitButler; the harness owns the agent (Assumption 2)
- **A new _agent_ UI, or a new app** â€” the agent-facing _enforcement_ stays CLI/action-boundary (no agent UI), and MGMT creates **no new app**: it is a **feature inside the existing SvelteKit desktop app** (`apps/desktop`), a settings-section _state_, not a new top-level route and not a new application
- **The non-git GitHub permission surface** (Actions, Secrets, Environments, Deployments, Webhooks, Pages, Dependabot, â€¦) â€” the POC scopes to the git + governance core
- **User-defined named permission templates** â€” beyond the built-in read/triage/write/maintain/admin presets, custom user-named bundles are deferred
- **Real-time session-permission refresh** â€” a permission change takes effect once committed to the target ref; live in-session refresh is deferred
