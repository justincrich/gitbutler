# Technical Requirements — Butler "Checks": a configurable, GitHub-Actions-like validations system

Constitution-layer engineering contract for the butler **validations / "Checks"** system — a GitHub-Actions-shaped *check* layer whose results gate the merge. The architecture is a **producer/consumer split**: the merge gate is a deterministic, **read-only CONSUMER** of a **signed, agent-unwritable check-result ledger**; butler ships a butler-controlled **DEFAULT EXECUTOR** as the **PRODUCER**. The hardness of forging a green check lives in *who produces the result and how it is bound* — **never** in the gate, which only reads. What the gate *proves* is narrow: *"the committed, target-ref-pinned check ran under the trusted executor and exited 0 at the current head OID"* — not *"the code is correct,"* and never *"non-fakeable."* This folder grounds every claim in the GitButler codebase scanned during design (`crates/but-authz/`, `crates/but-api/src/legacy/merge_gate.rs`, `crates/but-db/`, `crates/WORKSPACE_MODEL.md`, `crates/but/src/command/`); where a precise function or module is not yet verified against source, the file says so and marks it a planning-time confirmation rather than asserting it.

> **Two binding decisions baked into this folder (orchestrator, 2026-06-19):**
> - **DECISION A — the merge gate is READ-ONLY; `on-merge-attempt` is an orchestrator pre-merge step.** The gate never invokes the executor (it stays the pure, synchronous, read-only consumer `enforce_merge_gate` already is). When a check is `on-merge-attempt`, the **trusted CLI/daemon** runs `but check run` *immediately before* calling the merge action; the gate then consumes the current-head results. Produce and consume are distinct synchronous phases.
> - **DECISION B — naming avoids the `butler_actions` collision.** GitButler already has an "Actions" feature (the `butler_actions` macro table, the `actions::Platform` noun). This validations system uses **`but check`** (CLI), **`but-checks`** (crate), **`check_results`** (`but-db` table) — explicitly distinct (see `02-system-components.md` disambiguation). The committed config path `.gitbutler/actions/*.toml` is retained for now (confirm at planning whether to rename to `.gitbutler/checks/`).

> **This builds ON the governance PRD, not beside it.** The governance initiative (`.spec/prds/governance/`) already shipped the merge gate (`crates/but-api/src/legacy/merge_gate.rs:40` `enforce_merge_gate`), `but-authz` (the `Authority`/`AuthoritySet`/`Denial` engine), the ref-pinned `.gitbutler/*.toml` config convention, and the `local_review_verdicts` `but-db` table + its **forgeability lesson** (R6). Checks reuses **all** of it: the same `enforce_merge_gate` entry point gains a *read-only required-checks clause*; the executor's results live in a **new `but-db` ledger that learns from `local_review_verdicts`' forgeability** by being signed + producer-bound; the denial contract gains one code (`gate.check_required`) alongside the existing `gate.review_required`. The single most important inheritance is the governance posture's honesty discipline: **name what is bound vs. what is accepted-leak, and never present the gate as the source of forgery-hardness.**

## The producer/consumer thesis in one line

> **The merge gate is a deterministic, read-only consumer; the butler-controlled default executor is the producer; the hardness of forging a green check = who-produces (executor ≠ the gated agent) + how-bound (signed, SHA-bound, recorded in an agent-unwritable ledger) — never the gate. The gate proves only that the committed, ref-pinned check ran and exited 0 at the current head OID. Results are valid for the current head OID only and are invalidated on every graph/ref mutation; the gate-read→merge-commit window is not atomic (R12).**

## Routing & Views

**This slice adds NO new route and NO new app.** The Checks surface is backend + CLI: a NEW `but-checks` crate (executor + contract types), a NEW `but-db` ledger table, an EXTENDED `merge_gate.rs` clause inside `but-api`, and NEW `but check …` CLI verbs in the `but` crate. A management UI for check definitions (an admin-only settings section in `apps/desktop`, parallel to governance's MGMT group) is **deferred** — when added it will be a STATE of the existing Settings surface (a **Checks** tab inside governance's "Permissions & Governance" page + a REQUIRED CHECKS field in Branch Gates), not a new top-level route. The full frontend/UI contract — the v1 CLI dual-audience output design, the deferred GUI component inventory, the conclusion→token mapping, and the lite-is-React port caveat — lives in [`09-frontend-ui.md`](./09-frontend-ui.md). The product-manager-authored top-level PRD sections own the UX framing; this folder owns the engineering contract.
