# Capability Chains

Boundary-crossing flows where a promise must hold across hops. This initiative has two — the **produce → sign → record → consume** chain (where the producer/signature binding is the contract that makes forgery hard) and the **SHA-reset invalidation** chain (where keying-by-OID is the contract that prevents a stale check from passing a rewritten head). Both compose with the governance chains (`CAP-AUTHZ-01`, `CAP-CONFIG-01`) at the merge gate.

> **Naming:** crate `but-checks`, CLI noun `but check`, table `check_results` — distinct from the pre-existing `butler_actions` feature (see `02-system-components.md`).

> **DECISION A (read-only gate):** the merge gate is the pure, synchronous, **read-only** consumer that `enforce_merge_gate` (`merge_gate.rs:40`) already is — it **never invokes the executor**. `on-merge-attempt` checks are run by the **trusted CLI/daemon orchestrator** immediately before it calls the merge action (hop 2 below is an _orchestrator_ step, not a gate step); the gate then consumes the current-head rows (hop 6). Produce and consume are distinct synchronous phases.

## CAP-CHECK-01 — define → execute → sign → record → gate-consume (read-only) → STEER-on-fail

**Promise:** a merge is permitted only if every _required_ check name has a result whose `conclusion` satisfies (default: `success`), whose `signature` verifies against the trusted producer, and whose `head_oid` equals the **current** head OID — and that result was produced by the **trusted executor, not the gated agent**. Otherwise the merge is denied with the structured contract (`gate.check_required`, exit 1) and the agent is STEERed to run the check at head. The producer's recording is **deterministic engine code** invoked after a real run; the gate's evaluation is **deterministic and read-only** (it never runs a check and never consults the agent's textual claim).

| Hop | From → To                       | Boundary contract                                                                                                                                                                                                                                                                                                                                                             |
| --- | ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | author → check definition       | `but check define` writes `.gitbutler/actions/<name>.toml`; it is **inert** until committed to the target ref (ref-pinned) and is `administration:write`-gated (governance config gate, `config_mutate.rs:18`). Weakening a required check is subject to the **bootstrap invariant** (it must clear the pre-change required checks — R11/Stance 9)                            |
| 2   | trusted orchestrator → executor | the **trusted CLI/daemon** (NOT the gate, NOT the agent's process) runs `but check run` (statuses:write-gated trigger) for each `on-merge-attempt` check **immediately before calling merge** (DECISION A); `but_checks::executor::run_and_record` loads the definition **at the target ref** (the caller picks the _name_; it cannot supply or alter the _command_ — R1/R11) |
| 3   | executor → conclusion           | the executor materializes an isolated checkout at the head OID (R13), runs the check's `run` spec, and derives a typed `Conclusion` from the **actual** outcome (exit status) — never from an agent-supplied value (R6). v1 emits `{success, failure, timed_out}`; the gate blocks on any non-`success` required conclusion                                                   |
| 4   | executor → signature            | sign the canonical `(name, head_oid, conclusion)` tuple with the producer key from `but-secret`; the head_oid is the **current** head OID at run time                                                                                                                                                                                                                         |
| 5   | executor → ledger               | **deterministic engine recording** — insert the signed row into the `check_results` `but-db` table, keyed by `(target, name, head_oid)` (`head_oid` is the correctness key; `target` is a mutable name); always happens after a run, never skippable by the agent                                                                                                             |
| 6   | merge gate → ledger             | `enforce_merge_gate` (read-only) reads `check_results WHERE target=? AND head_oid=<current_head_oid>` (the SHA-reset read), **verifies each signature** (`but_checks::verify`), and calls the pure `required_checks_satisfied(...)`. It does NOT run any check — a missing current-head row yields `check_missing`, not an on-demand run                                      |
| 7   | gate → decision                 | `Ok(())` → the required-checks clause passes (composed with the existing `Merge` authority + review clauses); `Err` → `{error:{code:"gate.check_required", message, remediation_hint, unmet}}`, exit 1 → STEER                                                                                                                                                                |

- **Failure modes:** an agent-authored conclusion → ignored (the executor derives it from the run, R6); a feature-head edit to the check definition → ineffective (read at the target ref, R1/R11); a config change that weakens a required check → blocked by the pre-change required checks (bootstrap invariant, R11); a forged-unsigned ledger row → treated as absent (signature does not verify, R4); a forged-signed row via a leaked producer secret → **accepted residual** (R2 — the agent shares the OS user and can read the keyring secret in the personal-tenant model); a `failure`/`cancelled`/`timed_out` at head → `gate.check_required` (`check_failed`); a result only at an old head → `check_stale_at_head` (R3); a head advance inside the gate-read→merge-commit window → **not detected** (R12 TOCTOU, worst on the forge path); a malformed check/gate config → fail closed (`config.invalid`); **an N-API merge route that skips `but-api` → the clause is bypassed (R10, inherited governance R14)** — the promise holds only for callers that route through `but-api`/`enforce_merge_gate`.
- **Real-service proof:** integration test against **real `but-checks` + real `but-db` + real git** (the `crates/but-api/tests/merge_gate.rs` + `crates/but-authz/tests/` pattern):
  - (a) a required check with a signed `success` at head → merge allowed;
  - (b) a required check with `failure` at head → `gate.check_required`;
  - (c) a forged-unsigned `success` row → ignored, merge denied;
  - (d) the agent's textual "passed" claim with no ledger row → merge denied;
  - (e) **the gate runs nothing (DECISION A):** with a required check and NO current-head row, `enforce_merge_gate` returns `check_missing` and **does not invoke the executor** — assert no run/record side-effect occurs from the gate path (a new row must NOT appear after a gate evaluation);
  - (f) **bootstrap invariant (R11):** a merge that edits a required check's definition or the `[[required_check]]` set is itself blocked by the _pre-change_ required checks at the target ref;
  - (g) **no caller-supplied `Conclusion` (R6, NEGATIVE test — grep-assert alone is insufficient):** assert there is **no public `but-checks`-through-`but-api` function that accepts a caller-supplied `Conclusion` and records a satisfying row without a real run.** This is a _behavioral_ assertion, not just a `grep`: enumerate the `but-api` `legacy::checks` surface and confirm the only producing entry point is `run_check` (run-then-record); a test that attempts to record a `success` without a run MUST have no such API to call. (Flag for the criteria file: this negative integration test is a required acceptance criterion, because a grep for `record(conclusion)` can miss an aliased or re-exported path — R6.)
  - Build-gate grep-assert (supplementary, not sufficient on its own): the gate matches the `Conclusion` enum (not a string, R5) and there is no public `record(conclusion)` API (R6).
- **Owner:** `rust-implementer` (build) → `security-auditor` + `rust-reviewer` (verify).

## CAP-CHECK-02 — SHA-reset invalidation (a stale check never passes a rewritten head)

**Promise:** a check result is valid for the head OID it was produced against, and only that OID; when history is rewritten (rebase/amend/reorder/commit-edit), prior results become stale and cannot satisfy the gate at the new head — the executor must re-run. (The keying makes a _stale_ row non-satisfying; it does NOT make read-then-merge atomic — that gap is R12.)

| Hop | From → To            | Boundary contract                                                                                                                                                                                                                                                    |
| --- | -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | executor → ledger    | every recorded row carries `head_oid` = the head OID at run time (CAP-CHECK-01 hop 4/5)                                                                                                                                                                              |
| 2   | agent → graph editor | a history rewrite goes through `but_rebase::graph_rebase::Editor::rebase()` (`crates/WORKSPACE_MODEL.md`: graph editor for ref-placement mutations), which materializes **new commit OIDs**; `Editor::commit_mappings()` (`graph_rebase/mod.rs:479`) exposes old→new |
| 3   | repo → gate          | the current source head OID now differs from the OID prior rows were keyed to                                                                                                                                                                                        |
| 4   | gate → ledger read   | `enforce_merge_gate` (read-only) reads only rows where `head_oid == current_head_oid` (`current_head_oid`, `merge_gate.rs:78`); prior-OID rows are **invisible** — automatic staleness, no explicit invalidate write                                                 |
| 5   | gate → decision      | with no satisfying row at the new head, the required check surfaces `check_stale_at_head` / `check_missing` → `gate.check_required` until the executor re-runs at the new head                                                                                       |

- **Failure modes:** matching by check _name alone_ (ignoring `head_oid`) → a stale-head `success` passes a rewritten head (the R3 hole — Blocking); keying on `(target, name)` without `head_oid` → a row from a reused branch name at a different OID could leak (the `target`-is-mutable trap — `03-data-schema.md`; `head_oid` is the backstop); a head advance between the gate read and the merge commit → not detected (R12 TOCTOU); a re-run at the new head not producing a new row → the gate stays blocked (correct, fail-closed); an (optional) prune step deleting a still-reachable head's rows → a spurious re-run (housekeeping bug, not a soundness bug — the gate just re-blocks until re-run).
- **Real-service proof:** integration test (real git + real graph editor): produce a signed `success` at head `A`; rewrite history (amend/rebase via the editor) to head `B`; assert (i) the row at `A` no longer matches the gate's current-head read, (ii) the gate returns `gate.check_required` with `check_stale_at_head`, (iii) re-running the executor at `B` produces a new satisfying row and the gate then allows. This is the direct analogue of governance's stale-approval test (`review_requirement.rs` `approval_stale_at_head`).
- **Owner:** `rust-implementer` (build) → `security-auditor` + `rust-reviewer` (verify).

## Composition with the governance chains

Both Checks chains terminate at the **same read-only merge gate** the governance chains do, so the boundary triggers compose:

| Trigger                       | Chain                                                                           | Where it fires                                                                                                                                                                      |
| ----------------------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `merge`/`land`                | governance `CAP-AUTHZ-01` (authority) + Checks `CAP-CHECK-01` (required checks) | `enforce_merge_gate` — authority clause then **read-only** required-checks clause; the `on-merge-attempt` _run_ is sequenced by the orchestrator just before this call (DECISION A) |
| config-read-at-ref            | governance `CAP-CONFIG-01` + Checks CAP-CHECK-01 hop 1/2                        | the check definition + the `[[required_check]]` policy are read at the target ref by the same ref-pinned mechanism; weakening them is subject to the bootstrap invariant (R11)      |
| `BUT_AGENT_HANDLE` resolution | governance `CAP-AUTHZ-01` hop 1                                                 | the `statuses:write` trigger guard resolves the principal the same way (trigger-authority ≠ producer-key)                                                                           |
| head-OID keying               | Checks `CAP-CHECK-02`                                                           | the gate's current-head read (the SHA-reset invariant); the read-then-merge window is non-atomic (R12)                                                                              |
| the N-API entry-point audit   | governance R14 / Checks R10                                                     | the same audit protects the required-checks clause (it lives inside the audited `enforce_merge_gate`)                                                                               |

## Boundary-trigger scan

The Checks boundary triggers (`define` a check, `run`/execute via the orchestrator, `sign`, `record`, gate-consume-at-head read-only, head-OID keying, the SHA-reset rewrite, the producer-key read, the config-weakening bootstrap check) all map to one of the two chains above, both terminating at the governance merge gate. The one chain Checks adds beyond governance's is the **signing/producer-key chain** (hop 4 of CAP-CHECK-01 + the `but-secret` key read) — and its v1 strength (symmetric HMAC, agent-readable in the personal-tenant model) and deferred closure (Ed25519 + sandboxed executor) are named in `07-technical-risks.md` R2/R4. The newly-named non-atomic gate-read→merge-commit window (R12) and the clean-workspace-checkout cost (R13) are carried in the risks file. No remote-runner / lease / job-broker chain exists in this slice (no-broker v1, Stance 2). Capability-chain coverage is therefore **complete for the v1 scope.**
