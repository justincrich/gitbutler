# Technical Risks

Risks carried into implementation, with the mitigation each gets in this slice. Some are **accepted by design** (the v1 scope line, inherited from the governance posture); the rest are correctness risks the build must close. The honesty discipline is governance's: **name what is bound vs. what is accepted-leak, and never present the gate as the source of forgery-hardness.**

> **Naming:** crate `but-checks`, table `check_results`, CLI noun `but check` — distinct from the pre-existing `butler_actions` feature (see `02-system-components.md`).

**Risk register: 14 risks.** R1, R3, and **R14** are **Blocking** correctness risks (the executor-isolation invariant, the SHA-reset invariant, and the fail-open-via-protected-early-return control-flow hole — the properties the design exists to provide). R2, R4, and **R11 (now High)** are the **High** integrity risks (the signing-key root of trust, the ledger's v1 forgeability ceiling, and the config-bootstrap self-escalation). R7–R10 are inherited accepted-leak residuals from governance. **R12 (SHA-reset TOCTOU)**, **R13 (clean-workspace checkout in the one-worktree model)**, and **R14 (required-checks fail-open via the protected-branch early-return)** are newly-named butler-specific risks.

| # | Risk | Severity | Why it matters | Mitigation in this slice |
|---|---|---|---|---|
| R1 | **Executor-isolation strength** — can the agent influence the executor (so it produces a `success` for failing work)? | **Blocking** | The entire forgery-hardness claim rests on the executor being *the trusted process, not the agent*. If the agent can make the executor run a tampered command, skip the real check, or report `success` for a failing run, the gate is consuming a result the agent effectively authored | The executor runs **in the trusted daemon/CLI the human runs**, NOT the agent's process (Stance 1); it loads the check's `run` spec from `.gitbutler/actions/*.toml` **at the target ref** (ref-pinned — a feature head can't redefine the command to `true`), and the `Conclusion` is derived from the **actual** run outcome (exit status), not from anything the agent supplies. The build MUST assert (integration test) that the executor ignores agent-supplied conclusions and that a feature-head edit to the check definition does not change the command the gate's judgment runs. **Residual:** the agent can still influence the *working tree the check runs against* (it's a shared tree) — but that is the point of a check (run against the code), and a check run against agent-modified code that genuinely fails will conclude `failure`. The agent cannot make a *failing* run report `success`. (See also R13 — getting a clean checkout at the head OID in the one-worktree model is itself non-trivial.) |
| R2 | **Signing-key secrecy is the root of forgery-hardness** | **High** | The gate trusts a row iff its signature verifies. With **HMAC-SHA256 (symmetric v1)** the same repo-local producer secret signs and verifies; an agent that can **read** that secret can forge a correctly-signed `success` row directly in the ledger. **In the personal-tenant model the agent shares the OS user with the executor**, so it can read the `but-secret` keyring producer secret **WITHOUT elevated privilege** — this is a real, named path, not a hypothetical | v1 stores the producer secret via `but-secret` (keyring), not in the working tree or a committed file. This **RAISES THE BAR** from `local_review_verdicts`' zero integrity to "forging requires reading the producer secret" — a real improvement, **NOT a closed boundary, and explicitly NOT a closure of the agent-forgery path** in the personal-tenant model (the agent shares the OS user and can read the secret). **Named accepted residual:** an agent that reads the keyring secret can forge. **Deferred closure (the real fix):** (a) **Ed25519** (asymmetric — the agent holding the public key cannot forge), AND (b) an **OS-sandboxed executor** where even the private key is unreachable by the agent process. The build MUST NOT present HMAC-v1 as unforgeable or as "closing the agent-forgery path"; it is the corrected-but-not-final integrity layer (the governance R6 → HMAC → Ed25519 escalation, applied here). |
| R3 | **The SHA-reset race** — a result lands, then the head moves | **Blocking** | A `success` produced at head `A` must NOT satisfy the gate after the agent rewrites history to head `B` (the un-reviewed/un-checked new code). Matching by check *name alone* (ignoring `head_oid`) is the exact hole — it would let stale green checks pass a rewritten head | Every ledger row is keyed by `head_oid`; the gate reads **only** rows at the current source head OID (`current_head_oid`, `merge_gate.rs:78`) and matches `(name, head_oid == current_head_oid)`, never `name` alone (Stance 5). A rewrite via `but_rebase::graph_rebase::Editor` moves the OID, so prior rows are **automatically stale**. The build MUST include an integration test: produce a `success` at head `A`, advance the head (rebase/amend) to `B`, assert the gate returns `gate.check_required` with `check_stale_at_head` (the same shape as the review evaluator's `approval_stale_at_head`, `review_requirement.rs:14`). A gate that passes a stale-head check is not done regardless of green unit tests. **Note:** this invariant makes a *stale* row non-satisfying but does NOT make read-then-merge atomic — that is the separate R12 TOCTOU. |
| R4 | **Ledger forgeability ceiling (v1)** — "agent-unwritable" is an integrity claim, not a filesystem claim | **High** | "The agent can't write the ledger" is weak in a shared-filesystem POC (the SQLite file is reachable). Forgery-hardness must NOT rest on the agent lacking DB write access — it must rest on the signature | The ledger's defense is the **signature**, not write-protection: a direct INSERT without a verifying signature is treated as **absent** by the gate (Stance 3 / `03-data-schema.md`). This is the correction of governance R6 (where `local_review_verdicts` had *no* signature and a direct INSERT trivially satisfied the gate). **Residual (folds into R2):** with symmetric HMAC and a shared OS user (personal tenant), a forged-but-correctly-signed INSERT is possible because the agent can read the secret. The build MUST test the gate against a **forged-unsigned** row (asserting it's ignored) AND name the forged-signed-via-readable-secret path as the R2 residual. The build MUST NOT claim the ledger is tamper-proof. |
| R5 | **Stringly-typed conclusion** — a typo or unknown variant silently passes/fails | Medium | If `conclusion` is a `String` and the gate does `== "success"`, an unknown/misspelled token silently fails (or, worse, a permissive compare silently passes) | `Conclusion` is a Rust enum parsed from a fixed token set (the 6-variant *type*: `success`/`failure`/`neutral`/`cancelled`/`timed_out`/`skipped`); an unknown token is a **parse error at record time**, not a silent gate outcome (Stance 3). v1 only *produces* `{success, failure, timed_out}` (the rest are reserved for forward-compat/GH-import — `03-data-schema.md` Q3), but the gate blocks on any non-`success` required conclusion regardless. Grep-assert (reviewer): the gate's decision matches on the `Conclusion` enum, never a raw string. |
| R6 | **The "lying executor" if mis-scoped** — a producer that reports `success` without running the check | Medium | If the recording API accepted a caller-supplied conclusion, a mis-scoped or buggy caller could record `success` for work that never ran — re-introducing the very fakeability the design removes | Recording is **deterministic engine code inside `but-checks::executor::run_and_record`**: it derives the `Conclusion` from the actual run and signs it; there is **no public API that records a caller-supplied conclusion** (the `but-api` boundary exposes `run_check`, which runs-then-records, not `record(conclusion)`). The build MUST NOT add a `record_result(conclusion)` public entry point that bypasses the run. **Reviewer asserts the recording path always follows a real run — and `08-capability-chains.md` adds the integration-test requirement that NO public `but-checks`-through-`but-api` fn accepts a caller-supplied `Conclusion` without a real run (grep-assert alone is insufficient — see the negative integration test there).** |
| R7 | **Fork / untrusted-contributor checks are out of scope** | Accepted | Running an *untrusted* contributor's check (fork-PR) safely is a hard isolation problem (the check command could be malicious); v1's executor runs **trusted, repo-local** definitions only | **Out of scope for v1** (Stance 2). The v1 executor runs the repo's own committed, ref-pinned check definitions, produced by the trusted process — it does NOT execute an untrusted PR's check spec. Named as a v1 limitation; the deferred runner protocol + a sandboxed runner is the closure. The build MUST NOT claim v1 safely runs untrusted/fork checks. |
| R8 | **No broker → no remote/labelled runners (v1)** | Accepted | The executor runs in-process in the trusted daemon/CLI; there is no lease/long-poll/labelled-runner protocol, so checks cannot be dispatched to remote or pooled runners | **Accepted v1 scope** (Stance 2) — it makes the trust boundary trivial (the producer *is* the trusted process). The runner protocol is **deferred + named**. The build MUST NOT present v1 as supporting remote runners. |
| R9 | **The merge gate binds the governed `but` action only** (inherited governance R11) | Accepted | A merge done entirely on the forge (auto-merge / UI / CI) or by raw push to the protected ref does not pass through `enforce_merge_gate`, so **no local required-check is consulted** | Same accepted-leak class as governance R11 (the local trunk is immutable, `upstream_integration.rs:159`; landing is a forge op). The required-checks clause binds the **governed `but` merge**; the bypass paths are named, and the deferred server-side pre-receive (push) + forge-side required-checks (forge) are the closures. **This is also where R12's TOCTOU is worst** (a forge-side head advance after the local gate accepted). The build MUST NOT claim the required-checks clause binds forge/push merges. |
| R10 | **N-API / direct-caller bypass of the gate** (inherited governance R14) | Medium (Accepted residual) | The required-checks clause lives inside `enforce_merge_gate`, which only governs callers that route through `but-api`. An un-audited `but-napi` merge route reaching a lower-level crate directly skips the clause — same hole class as governance R14 | Inherited from governance unchanged: the N-API audit (governance T-AUTHZ-016b) that routes every consequential `but-napi` route through `but-api` also covers the required-checks clause, because the clause is inside the same `enforce_merge_gate` the audit protects. **Accepted residual** until that audit; the steel-trap is the eventual closure. The build MUST NOT claim the clause binds N-API callers it has not audited. |
| R11 | **Config-bootstrap self-escalation via the check definition / required-check set** | **High** (escalated from the prior register) | A green check proves only *the committed check ran* — so the cheap bypass is not forging a result but **weakening the required-check set**: an `administration:write` holder could rewrite `.gitbutler/actions/cargo-test.toml` to `run = "true"` (always passes) or drop a name from `[[required_check]]`, defeating the gate's judgment. If a config change could land under the *weakened* checks it introduces, the gate is trivially defeatable | **Closure = the bootstrap invariant (the real fix, Stance 9 + `03-data-schema.md`):** a change that modifies a required check's definition (`.gitbutler/actions/*.toml`) or the `[[required_check]]` set **MUST itself clear the currently-required checks at the target ref before it can land** — the required-check config is **self-protecting** (same pattern as governance's ref-pin self-escalation prevention). The pre-change required set (read at the target ref) governs the config-change merge, so you cannot weaken the checks that judge a merge in that same merge. Ref-pinning alone (feature-head inertness, `merge_gate.rs:211` pattern) is **necessary but NOT sufficient** — the build MUST implement the bootstrap invariant and MUST NOT present feature-head inertness as the closure. **Named accepted residual (parallel to governance R10):** a same-ref admin who *also* reaches the fence remains a named residual; config-touching changes are reviewable as a diff. The build MUST include an integration test: a merge that weakens a required check is itself blocked by the pre-change required checks. |
| R12 | **SHA-reset TOCTOU** — gate-read and merge-commit are not atomic | **High** | The gate resolves `current_head_oid` and accepts the merge in two steps; the head can **advance between them**. A check `success` is valid at the OID the gate read, but by the time the merge commits, the head may have moved to un-checked code. This is **worse on the forge-merge accepted-leak path (R9)**: the local gate accepts a `success` at OID `A`, but the actual forge merge lands at a later OID `B` (the agent pushed in the window), so the gate's judgment applied to code the check never saw | **Not closed in v1 — named residual.** The SHA-reset invariant (R3/Stance 5) makes a *stale* row non-satisfying, but it does NOT make read-then-merge a single atomic operation; a head advance inside the gate-read→merge-commit window is not detected. **Deferred closure:** (a) **"require up to date"** — re-resolve the head OID at merge-commit time and refuse if it changed since the gate read (a cheap local guard worth considering even in v1 for the local path), and (b) a **merge-queue** that serializes merge + head-advance. The build MUST name this as an open, non-atomic window — especially on the forge path — and MUST NOT present the SHA-reset invariant as eliminating the TOCTOU. |
| R13 | **Clean-workspace checkout at the head OID in the one-worktree model** | Medium (planning risk / hidden cost) | The executor must run each check against the code **at the exact current head OID**. GitButler is virtual-branches-over-ONE-worktree: the single shared worktree is where the agent is actively editing, may be dirty, and is a *workspace* projection of several virtual branches — not a clean checkout of one branch's head OID. "Just run the command in the repo dir" would run against the agent's live, possibly-dirty tree, silently breaking the head-OID binding (the result would be signed for an OID whose code was not what ran) | **Under-specified — flagged as a planning risk (see `02-system-components.md`).** The executor MUST obtain an **isolated checkout at the head OID** without disturbing the shared worktree the agent uses (candidates: a throwaway `gix`/temp checkout from the object DB at the OID; `git worktree add --detach <oid>` at a temp path; a worktree-free object check for purely-git checks). The planner MUST resolve: how the isolated checkout is materialized, its cost/latency on every `on-merge-attempt` run, interaction with the shared index/locks (`BUT_WS_LOCK_DEBUG`), and the hard constraint that a check run **MUST NOT mutate or contend on the agent's shared worktree**. The build MUST NOT assume the worktree == the head OID. |
| R14 | **Required-checks fail-open via the protected-branch early-return** | **Blocking** | `enforce_merge_gate` returns `Ok` early for a target branch that is not flagged `protected` (`merge_gate.rs:50-56`). If the required-checks clause is appended *after* that guard, a branch carrying a `[[required_check]]` set but **not** marked `protected` merges with **zero** check enforcement — a silent fail-open, the inverse of the gate's whole purpose. Governance only ever gated *protected* branches; required checks must not inherit that as a bypass | **Closure (mandatory, `04-api-design.md`):** read the required-set **before** the early-return and take the early-return **only** when the branch is neither `protected` **nor** carries required checks; clause 3 then runs whenever `required` is non-empty, **independent of the `protected` flag**. The loader reads `[[required_check]]` independent of protection; a `[[required_check]]` on a branch the gate cannot otherwise enforce is `config.invalid` (fail-closed), never silently un-enforced. The build MUST include an integration test: a branch with a required check but **not** flagged `protected` still blocks on that check (the clause is not bypassed by the early-return). |

## The risks the reviewer must treat as honesty tests
- **R1 (executor isolation) and R3 (SHA-reset) are the two invariants the design exists to provide.** A build that lets the agent influence the executor's conclusion, or that passes a stale-head `success`, has not built Checks regardless of green tests. These are Blocking, with mandatory integration tests against real `but-checks` + real `but-db` + real git.
- **R2/R4 (signing-key + ledger integrity) must stay named, never quietly "mitigated" into looking closed.** The v1 HMAC layer corrects governance R6's *zero* integrity but is *symmetric*, and in the personal-tenant model the agent shares the OS user and can read the producer secret — so it RAISES the bar, it does NOT close the agent-forgery path. Presenting the v1 ledger as tamper-proof, or HMAC as "closing forgery", is the same misrepresentation class as governance R1. The Ed25519 + sandboxed-executor closure is deferred and must be named as such.
- **R11 (config-bootstrap self-escalation) is High and its closure is the bootstrap invariant, not ref-pinning alone.** A build that relies only on feature-head inertness, leaving the "weaken the check then merge under the weakened check" path open, has not closed R11. The self-protecting required-check config is mandatory.
- **R12 (SHA-reset TOCTOU) is an open, non-atomic window** — naming it (and noting it is worst on the forge path) is required; presenting the SHA-reset invariant as atomic is a misrepresentation.
- **R13 (clean-workspace checkout) is an under-specified hidden cost** — the planner must resolve how the executor gets an isolated head-OID checkout in the one-worktree model; assuming the worktree is the head OID silently breaks the head-OID binding.
- **R14 (fail-open via the protected early-return) is Blocking control flow.** A build that appends the required-checks clause *after* the `protected()` early-return — so a branch carrying required checks but not flagged `protected` merges unchecked — has not built the gate. The required-set MUST be consulted before the early-return, and clause 3 MUST run independent of `protected`.
- **R7/R8 (fork-checks + no-broker) are accepted v1 scope, not closed boundaries.** v1 runs trusted, repo-local checks in-process. Presenting v1 as safely running untrusted/fork checks, or as supporting remote runners, is a misrepresentation.
- **R9/R10 (forge/push + N-API bypass) are inherited governance residuals.** The clause binds the governed `but` merge only; presenting it as binding forge/push/N-API merges it has not audited is the same misrepresentation class as governance R1.
- **R5/R6 are grep-asserted invariants — plus R6 carries a negative integration test.** If the gate's decision keys off a raw conclusion string (R5), or a public API records a caller-supplied conclusion without a run (R6), the slice is not done regardless of green tests. R6's grep-assert is backed by the `08-capability-chains.md` integration test that no public `but-checks`-through-`but-api` fn accepts a caller-supplied `Conclusion`.

## Relationship to the governance risk register
Checks does **not** re-litigate governance's accepted leaks (R1 fence, R2 identity, R11 forge-merge, R14 N-API, R12 human-as-owner, R13 direct-file-edit) — it **inherits** them, because it extends the same gate. R9 and R10 above are the two governance residuals that touch the merge boundary directly and so are re-stated for clarity. The **new** risks Checks introduces are R1–R6 + R11 + R12 + R13 + R14 (the executor, the signing key, the SHA-reset binding of *check* results, the typed conclusion, the lying-executor guard, the check-definition/required-set bootstrap, the SHA-reset TOCTOU, the clean-workspace checkout, and the protected-early-return fail-open). The build MUST treat the governance residuals as still-open and Checks' new invariants (R1, R3, R14) as Blocking.
