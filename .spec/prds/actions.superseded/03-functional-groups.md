# Functional Groups

| Group                      | Prefix | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------------------------- | ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------- | ------- | --------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Action Definition          | DEFN   | Validations are **config-as-code**: a list of **named checks** in committed, ref-pinned `.gitbutler/actions/*.toml`, each = a `name` (stable identity, the analog of a GitHub status `context` / check-run name) + a `trigger` (`on-commit` / `on-merge-attempt`) + a `run-spec` (a **command** or a repo `./path` **script** — local-action resolution only in v1) + a `required` flag + a success mapping (process exit `0` → conclusion `success`). The config is read at the **target ref** when the gate evaluates required checks (exactly like governance's `gates.toml`), so an agent **cannot weaken, drop, or flip `required` on a required check in its own change** to slip a merge past the gate. `uses`-registries, JS/Docker/composite types, matrix/DAG/reusable workflows, and the `${{ }}` expression engine are deferred.                                                                                                                                                                                                                                                                                                                               |
| Check Executor             | EXEC   | The butler-controlled **default executor** that **produces** check results: it runs a defined check against a **checked-out head SHA in a clean workspace**, captures stdout/stderr + the exit code, and maps it to a typed conclusion. **Executor ≠ the gated agent** — the executor runs in the **trusted daemon/CLI the human runs, NOT the agent's process or a surface the agent controls** (whoever runs the check controls the result). It performs **real execution — no stubbed success, ever** — and injects only **minimal, executor-isolated** butler secrets the check declares. It produces the inputs the ledger signs; it does **not** adjudicate the gate (separation of who-executes from who-adjudicates). A detached runner lease/long-poll/label protocol + autoscaling is deferred.                                                                                                                                                                                                                                                                                                                                                                  |
| Check-Result Ledger        | LEDG   | **Signed `(name, head_sha, conclusion, producer-identity)`** records on an **agent-unwritable** recording surface, written by **deterministic engine code** (recording a result is never an agent decision — the deterministic-vs-probabilistic doctrine). Identity = **`(name, head_sha)`**; the gate reads results for the **current** head SHA only. The **SHA-reset invariant** is load-bearing: a result is valid for the head SHA it was produced against and is **invalidated/recomputed on every graph/ref mutation** (GitButler rebases/edits/reorders commits constantly via the rebase editor) — **a green bound to an ancestor SHA never satisfies the gate**. The conclusion is a typed terminal verdict (`success                                                                                                                                                                                                                                                                                                                                                                                                                                            | failure | neutral | cancelled | timed_out | skipped`) distinct from a non-terminal status, mirroring GitHub so policy semantics port. This is the **hardened** counterpart to governance's deliberately-forgeable review store (its R6) — a machine verdict for an agent with no out-of-band stakes cannot be left forgeable. |
| Required-Checks Merge Gate | GATE   | A **required-checks clause** that **consumes** the ledger: a change may merge into the target **iff** for **every required check name** (named by the `[[required_check]]` policy in the target-ref `gates.toml`, mirroring governance's `[[gate]]` review requirement) there is a **signed `success` bound to the current head SHA**. It **composes** with the governance merge gate — evaluated by the same deterministic on-action code as `merge`-authority and the review requirement, at the governed `but` merge boundary — and it **fails closed** on a required check that is missing, unreadable, stale (ancestor-SHA), unverifiable (bad/absent signature), or concluded `failure`/`timed_out`/`cancelled`. On a miss it returns the governance-consistent `{code, message, remediation_hint}` naming the unmet check(s) and the miss-reason (`check_missing` / `check_failed` / `check_stale_at_head` / `check_unverifiable`) plus **STEER** fields, so the orchestrator can **redirect** the agent to run/fix the right check. It is a **deterministic consumer** — it never runs a check, never trusts an agent's "tests pass," and never writes the ledger. |

## Use Case Summary

| Group                      | Prefix | UCs    | ACs     |
| -------------------------- | ------ | ------ | ------- |
| Action Definition          | DEFN   | 5      | 27      |
| Check Executor             | EXEC   | 5      | 30      |
| Check-Result Ledger        | LEDG   | 4      | 23      |
| Required-Checks Merge Gate | GATE   | 5      | 30      |
| **Total**                  |        | **19** | **110** |

Prefixes DEFN / EXEC / LEDG / GATE are distinct and unambiguous within this PRD. 19 UCs is within the 3–25 budget; every group has ≥1 UC (DEFN/EXEC/GATE 5 each, LEDG 4). DEFN establishes the config-as-code named-check schema (incl. the ref-pin that prevents self-weakening); EXEC is the non-fakeable producer (executor ≠ agent, real execution, isolated secrets); LEDG is the signed, agent-unwritable contract with the load-bearing SHA-reset invariant; GATE is the deterministic required-checks consumer that composes with the governance merge gate and emits the STEER redirect. **110 ACs / 112 e2e criteria** (110 per-AC 1:1 + 2 additive full-loop [human-gate] demos) — see [10-e2e-testing-criteria.md](./10-e2e-testing-criteria.md). The pipeline is: **DEFN defines → EXEC produces → LEDG records (SHA-bound, signed) → GATE consumes (composes with governance) → STEER redirects.** The 09-technical-requirements/ folder (authored separately by a rust-planner) grounds the crate/component contract.
