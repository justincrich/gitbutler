# Scope

**Scope Posture**: Full feature (default — a complete, fully-polished v1 slice).

This PRD is the **agent-non-forgeable validations slice** that turns GitButler into a quality-enforcement layer for orchestrated agents. The boundary sentence for every item: **Actions makes a change unmergeable until every named required check reports a signed `success` bound to the current head SHA, produced by a butler-controlled executor that is not the agent, recorded in an agent-unwritable ledger, and invalidated on every SHA-changing graph mutation.** It is the verification PRODUCER half; the governance merge gate is the CONSUMER (a new required-checks clause); STEER is the redirect. Everything opinionated about _which_ checks run lives in committed, ref-pinned config; the engine ships the mechanism only — a real executor, a signed ledger, and a deterministic gate clause. It is native to GitButler and grounded in GitButler's own crates and graph model. The CLI surface for this system is the **`but check`** noun (distinct from the existing `butler_actions` macro feature — see [00-overview.md](./00-overview.md) naming disambiguation).

## The guarantee is bounded (constraint, not a feature)

What the gate proves is **precise**: that the committed, target-ref-pinned check definition **actually ran by the trusted executor and exited `0`** against the exact head SHA being landed — i.e. the system is **non-fakeable by the agent for a fixed check definition**, not "non-fakeable" in any semantic-quality sense. The strength of the guarantee is exactly the strength of the committed check set; a weak check yields a weak guarantee. The engine protects _provenance and binding_ (who ran the check, against what code, authored by the trusted producer), never the semantic goodness of the check itself.

## The composition assumption (constraint, not a feature)

Actions does **not** stand alone, and it does **not** re-invent the governance gate. The required-checks clause **extends the existing governance merge gate** (`.spec/prds/governance/`), evaluated as part of the same deterministic on-action code, alongside the existing `merge`-authority and review clauses. Actions reuses governance's denial contract (`{code, message, remediation_hint}` + STEER fields), its ref-pin discipline (config read at the target ref), and its action boundary (`but-api` / the governed `but` merge). This PRD specifies the producer (executor), the contract (ledger), and the new clause (gate); it relies on governance for the surrounding gate machinery and for the process clauses it composes with.

## The human-in-the-loop constraint for required-check config changes

A change to the **required-check set itself** (adding/removing a check, flipping `required`, or editing a required check's `run-spec`) is a **policy** change, not a code change, and v1 does not trust a fully-automated agent-review pipeline to catch its implication. In an all-agent review loop, an agent reviewer may approve a diff that flips `required: true → false` or swaps a real check's `run-spec` for a trivially-passing command without recognizing it weakens the gate. Two protections apply, and both are constraints of this slice:

- **Engine-enforced (the bootstrap-invariant):** a change that modifies the required-check set or a required check's definition must itself clear the **currently-required** checks at the target ref before it can land (the required-check config is self-protecting — see [UC-DEFN-04](./04-uc-defn.md) and [UC-GATE-05](./07-uc-gate.md)). This is the same self-escalation-prevention shape governance uses for its ref-pin: you cannot use a weakening change to weaken the gate that judges that very change.
- **Process-required (human in the loop):** because an automated agent reviewer may not catch the policy implication of a `required:false` / required-set change, v1 **requires a human in the loop** to approve a change to the required-check configuration (the diff to the `[[required_check]]` policy in `gates.toml`, or to a required check's `run-spec` in `.gitbutler/actions/*.toml`). This is named here as a v1 constraint, not an engine-enforced gate beyond the bootstrap-invariant; richer policy presets are deferred.

## In Scope

### Action definition (config-as-code) — DEFN

- **Named checks** in committed, ref-pinned config (`.gitbutler/actions/*.toml`): each check = a `name` (stable identity, the analog of a GitHub status `context` / check-run name) + a `trigger` + a `run-spec` + a `required` flag + a success mapping
- **Triggers** restricted to two: `on-commit` (the check is eligible to run when a commit is created) and `on-merge-attempt` (the trusted CLI/daemon runs the required checks immediately before a governed merge, then the gate evaluates required checks at head). No cron / dispatch / external event in v1
- **Run-spec**: a **command** string **or** a repo-local `./path` script — **local-action resolution only** in v1, executed by the butler executor. No `uses` registry, no marketplace, no remote-ref action resolution
- **Success mapping**: a process exit-code mapping (exit `0` → conclusion `success`; non-zero → `failure`), so the conclusion is a deterministic function of the executed check, not an agent's claim
- **Required-set policy** (`[[required_check]]` in `gates.toml`, read at the target ref, mirroring governance's `[[gate]]` review requirement): names which defined checks are mandatory for a branch's merge gate. A check definition's `required` field declares its **default** requiredness; the **authoritative required-set the gate enforces** is this `[[required_check]]` policy. A defined check not in the required-set runs and records but does not block
- **Ref-pinned + committed**, read at the **target ref** (exactly like governance's `gates.toml`) when the gate evaluates required checks — so an agent cannot weaken, drop, or flip `required: false` on a required check **in its own change** to slip a merge past the gate
- **Self-protecting required-check set (the bootstrap-invariant)**: a change that edits the required-check set or a required check's definition must itself clear the currently-required checks at the target ref before it can land — the config that defines "good" is governed by the very checks it currently mandates
- The check config is **agent-editable as a file** but, like all governed config, weakening it is **inert until committed to the target ref**; touching it is a reviewable diff and (via governance) gated by `administration:write`, and a required-set change additionally requires a human in the loop (above)

### Check executor (the producer) — EXEC

- A butler-controlled **default executor** that runs a defined check against a **checked-out head SHA in a clean workspace**, captures stdout/stderr + the exit code, and maps it to a typed conclusion
- **Executor ≠ the gated agent**: the executor runs in the **trusted daemon/CLI the human runs**, NOT in the agent's process or on a surface the agent controls — the load-bearing anti-cheat (whoever runs the check controls the result)
- **Real execution, no stubbed success — ever**: the executor actually invokes the command/script; a stubbed pass, a fake conclusion, or a skipped run that records `success` is the cardinal sin, not a v1 simplification
- **Timeout handling**: a check that exceeds its declared `timeout_secs` is terminated and concluded `timed_out` (a blocking conclusion), never left to hang or silently recorded green
- **Concurrency-safe recording**: multiple checks running for a head (and across heads) produce/record results without clobbering one another; results are keyed by `(name, head_sha)` so concurrent producers do not overwrite each other's records
- **Minimal, executor-isolated secrets**: the executor injects only the minimal butler-held secrets a check declares it needs, scoped so the gated agent's check cannot read them out-of-band through the executor (a clean-workspace + injected-env boundary, the personal-tenant/internal-agent trust model — not a hostile-code sandbox in v1)
- **Captured output is masked before storage**: stdout/stderr persisted as ledger `metadata` is **agent-readable** (same OS user), so any injected-secret values must be masked before storage and raw-output retention is limited — metadata storage is not presented as safe with secret-injecting checks
- The executor produces the inputs the ledger signs; it does **not** itself decide the gate (separation of who-executes from who-adjudicates)

### Check-result ledger (the contract) — LEDG

- **Signed records** of `(name, head_sha, conclusion, producer-identity)` — the producer signs the tuple with a key the gated agent does not hold, so a result cannot be forged, replayed from another SHA, or hand-edited
- **Agent-unwritable recording surface**: results live in a protected store / server-validated oplog the gated agent cannot write to with producer authority; **recording a result is deterministic engine code**, never an agent decision (honoring the deterministic-vs-probabilistic doctrine)
- **Identity = `(name, head_sha)`**: a result attests exactly one check against exactly one commit; the gate reads results for the **current** head SHA only
- **SHA-reset invariant**: a result is valid for the head SHA it was produced against and is **invalidated/recomputed on every graph/ref mutation** (GitButler rebases/edits/reorders commits constantly via the rebase editor) — **a green bound to an ancestor SHA never satisfies the gate**. This is GitHub's "new push invalidates" property, made more load-bearing because butler changes SHAs routinely; the gate keys on commit IDs at the boundary (per `crates/WORKSPACE_MODEL.md`)
- **Metadata is agent-readable, masked, retention-limited**: the `metadata` column (stdout/stderr/log refs) is not part of the trust boundary and is readable by the agent (same OS user); injected secrets are masked before storage and raw-output retention is bounded
- **Conclusion is a typed terminal verdict** parsed/stored from the GitHub-compatible vocabulary (`success | failure | neutral | cancelled | timed_out | skipped`) distinct from a non-terminal status, mirroring GitHub so policy semantics port. In v1 the exit-code executor **produces** only `{success, failure}` (plus `timed_out` when the timeout wrapper fires); `{cancelled, neutral, skipped}` are **reserved/parsed-not-produced** (stored if a future producer emits them, but the v1 default executor does not author them)

### Required-checks merge gate (the consumer) — GATE

- A **required-checks clause** on the governance merge gate: a change may merge into the target **iff** for **every required check name** (named by the target-ref `[[required_check]]` policy in `gates.toml`) there exists a **signed `success` bound to the current head SHA**
- **On-merge-attempt is a pre-merge run + read-only gate**: the trusted CLI/daemon runs the required `on-merge-attempt` checks **immediately before** the governed merge, then the gate **consumes** the current-head results. The gate itself is a **pure read-only consumer — it never runs a check**; a required check that is stale at head makes the gate BLOCK with `check_stale_at_head` and STEER ("run the check"), it does not auto-re-run inside the gate
- **Composes** with the existing governance clauses — it is evaluated by the same deterministic on-action code as `merge`-authority and the review requirement, at the governed `but` merge boundary; it does not replace them
- **Self-protecting against config-weakening**: because both the governance config and the Action config are read at the target ref, and the bootstrap-invariant requires a required-set change to clear the currently-required checks, a change cannot weaken the gate that judges it
- **Fail-closed**: a required check that is **missing entirely** (never produced), **unreadable**, **stale** (bound to an ancestor SHA), **unverifiable** (bad/absent signature), or whose conclusion is `failure`/`timed_out`/`cancelled` ⇒ **merge blocked**, never vacuously satisfied; a **malformed Action config** at the target ref denies with `config.invalid` (fail-closed, mirroring governance), never an empty/satisfied required-set
- **Denial + STEER**: on a miss, the gate returns the governance-consistent `{code, message, remediation_hint}` naming the unmet check(s) and distinguishing miss-reasons (`check_missing` / `check_failed` / `check_stale_at_head` / `check_unverifiable` / `config.invalid`), plus STEER fields, so the orchestrator can redirect the agent to run/fix the right check
- **Deterministic consumer only**: the gate never runs a check, never trusts an agent's textual "tests pass," and never writes the ledger — it reads and adjudicates

## Out of Scope

### Deferred to later Actions layers (named, not denied)

- **A detached runner protocol** — a runner **lease / long-poll / labels / runner-groups** model and **autoscaling** (ARC-style ephemeral workers). v1 runs checks in the trusted local daemon/CLI; the detached-worker protocol is required only when execution must go **remote or untrusted-isolated**, and is explicitly future work
- **OS-sandboxed executor + Ed25519 signing (the agent-forgery closure)** — v1's HMAC + clean-workspace executor _raises_ the forgery bar but, in the personal-tenant model where the agent shares the OS user with the executor, does not fully _close_ agent forgery (the agent can read the keyring producer secret without elevated privilege). The deferred closure is an OS-sandboxed executor (the agent cannot read the producer secret) **plus** Ed25519 asymmetric signing (the agent never holds a signing-capable key). Named, not claimed closed in v1
- **Action types beyond local** — **JavaScript**, **Docker container**, and **composite** action types, plus a **`uses` registry / marketplace** and remote-ref (`owner/repo@sha`) action resolution. v1 is a command or a repo `./path` script only
- **Workflow orchestration** — **matrix / strategy** (include/exclude, fail-fast, max-parallel), a job **DAG (`needs`)**, and **reusable workflows** (`workflow_call`). v1 is a flat list of named checks
- **A full expression engine** — the `${{ }}` contexts/expressions/functions (github/env/steps/needs/matrix, `hashFiles`, `fromJSON`, status functions). v1 maps a command's exit code to a conclusion; no expression evaluation
- **Artifacts & caching** — `upload`/`download-artifact` and a build cache. v1 captures a conclusion (+ minimal evidence), not artifacts
- **OIDC / fine-grained external creds** — short-lived cloud-credential minting for checks that touch external systems. v1 injects only the minimal butler secret store
- **Fork / untrusted-contribution isolation** — the `pull_request` vs `pull_request_target` split and ephemeral hostile-code sandboxing. v1 assumes an internal-agent / personal-tenant trust model; gating untrusted external contributors is future work
- **Merge-queue / "require up to date" (the SHA-reset-TOCTOU closure)** — serialized landing and a "head must be at/after target tip" requirement (which forces a re-run on the rebased head). v1 evaluates required checks at the current head only; closing the gate-eval-vs-merge race is deferred (see Known Limitations)
- **Rich result UX** — annotations (file+line), live log-streaming, and a results dashboard. v1 captures the gate-critical conclusion (+ a summary/log link); rich `output` is deferred
- **Pluggable external producers** — consuming third-party check results posted by external CI. v1 ships the butler default executor as the sole producer; the ledger contract is designed to admit external producers later
- **Richer required-check policy** — required-by-default presets, ruleset-style composition, and machine-detectable "this diff weakens the gate" classification (so the human-in-loop requirement could be relaxed). v1 uses the explicit `required` flag + the bootstrap-invariant + a human in the loop for required-set changes

### Genuinely out (non-goals)

- **Certifying semantic code quality** — Actions proves the committed check ran and exited 0, never that the code is _correct_ or the check is a _good_ test; a vacuous check yields a vacuous guarantee (named in "The guarantee is bounded")
- **Driving the agent's reasoning** — Actions governs _whether a validated change may land_, never the agent's prompts, turn loop, or methodology (the harness owns the agent, per governance Assumption 2)
- **Closing governance's fence leak** — Actions hardens the _verification verdict's provenance_; it does not make raw `git` / `--no-verify` / an ungoverned direct N-API call impossible (those still bypass the governed `but` merge, the same accepted-leak class as governance R1). The required-checks clause binds the **governed** merge only
- **A new agent UI or a new app** — the agent-facing surface stays CLI/action-boundary (`but check ...`); v1 ships no Actions dashboard. (A human management surface, if any, would extend governance's existing MGMT settings section — out of this slice)
- **Replacing GitHub Actions / running real CI workloads at scale** — this is a local merge-gate validation system, not a hosted CI platform; scale/runner concerns are explicitly the deferred runner protocol
