# Roles

This initiative draws one decisive **trust boundary**: between the entity that **is gated** (the coding agent whose change must be proven to have run its checks) and the entity that **produces the proof** (the butler executor the human runs). The whole anti-cheat rests on those never being the same principal — _whoever runs the check controls the result_, so the gated agent must never produce its own passing verdict. Every role below is described by **what it may produce, write, and adjudicate**, because that is what makes the verdict **agent-non-forgeable for a fixed check definition** (the engine protects provenance + binding, not the semantic goodness of the check — see [00-overview.md](./00-overview.md)).

| Role                                                     | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Human fleet-owner / admin**                            | The human who **defines checks** (authors `.gitbutler/actions/*.toml` via `but check define`), **marks checks `required`**, and owns the executor's signing root. In the governance composition they are also the maintainer/administrator (`merge` + `administration:write`); changing the check config is an `administration:write`-gated, ref-pinned, governed change, and a change to the **required-check set** additionally requires a **human in the loop** to approve (an automated agent reviewer may not catch the policy implication — see [01-scope.md](./01-scope.md)). They are the **trust root** for verification — the executor runs as them (on their machine, their keys, personal-tenant trust), and the agent cannot reach this root. They decide _which_ validations define "good code"; the engine enforces _that_ those validations actually ran and passed (not _that_ the code is semantically good).                                                                                                                                                                   |
| **Implementer / reviewer agents (the gated principals)** | The LLM workers the orchestrator runs — the **subjects of checks**. They produce the _code_ a check validates, but they **cannot produce, sign, write, or weaken** a passing check result for a fixed check definition. An implementer cannot land a change until its required checks are green at head; a denied agent receives the structured `{code, message, remediation_hint}` + STEER fields naming the failed/missing check so it can fix and re-run. The agent runs `but` actions; it does **not** run the executor and has **no** write authority over the ledger. Caveat named in scope: in the personal-tenant model the agent shares the OS user with the executor, so v1 HMAC _raises_ the forgery bar but does not fully _close_ it (OS-sandbox + Ed25519 is the deferred closure). "Implementer"/"reviewer" are governance bundle labels, not check-enforcement keys.                                                                                                                                                                                                              |
| **The butler executor (the trusted producer)**           | Not a principal the agent can assume — the **butler-controlled process that runs checks**. It runs in the **trusted daemon/CLI the human runs**, checks out the head SHA into a clean workspace, **actually executes** the defined command/script (no stubbed success), enforces the check's `timeout_secs` (a timeout → `timed_out`, never a hang or a silent green), runs/records concurrently without clobbering (`(name, head_sha)`-keyed), injects only the minimal executor-isolated secrets the check declares (and masks injected-secret values before any captured output is persisted), captures the exit code → conclusion, and signs `(name, head_sha, conclusion, producer-identity)` with a key the agent does not hold. It is also what runs the required `on-merge-attempt` checks **immediately before** a governed merge (the pre-merge run), so the gate has current-head results to consume. It **produces** the inputs; it does **not** adjudicate the gate. **Executor ≠ the gated agent** is the single most important fact in this PRD.                                   |
| **GitButler engine (the deterministic recorder + gate)** | Not a principal — the **policy-enforcement system itself**, split across two deterministic responsibilities. As the **recorder**, it writes the signed result to the agent-unwritable ledger as engine code (never an agent decision) and **invalidates results on every SHA-changing graph mutation** (the SHA-reset invariant). As the **gate**, it evaluates the required-checks clause at the governed merge boundary — reading the ledger for the **current head SHA**, verifying each required check's signature + SHA-binding + `success`, and blocking otherwise — composing with the governance `merge`-authority and review clauses. The gate is a **pure read-only consumer**: it **never runs a check** (a check stale at head makes it BLOCK with `check_stale_at_head` + STEER "run the check", it does not auto-re-run inside the gate). It also enforces the **bootstrap-invariant** — a change that edits the required-check set must clear the currently-required checks at the target ref. It is the enforcer of every "cannot" in this PRD and never trusts an agent's claim. |
| **The orchestrator (coding harness)**                    | An external harness (Claude Code, Codex, a ralph loop) that dispatches the agents and drives GitButler via the `but` CLI. **Trusted by outcomes, not policed** (the same trust class as in governance — your machine, your keys, your reasoning loop). It is bound only by the gate's outcome: it cannot land a change whose required checks are not green at head. It **consumes the STEER signal** — on a denial, it re-routes the agent to run/fix the named check (including running a stale-at-head check via `but check run`). It does not produce check results and is not the executor.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

## The producer-vs-adjudicator separation (the load-bearing split)

GitButler's deterministic-vs-probabilistic doctrine maps cleanly onto two distinct deterministic responsibilities here, and they are deliberately **not** the same component:

- **Produce** (the executor): runs the real check, captures the real conclusion, signs it. This is butler-controlled code, never the gated agent. It is also the principal that runs the `on-merge-attempt` checks in the pre-merge step. The output is a signed `(name, head_sha, conclusion, producer-identity)` tuple.
- **Adjudicate** (the engine gate): reads the ledger, verifies signature + SHA-binding + `success` for every required check at the current head, and decides merge / block. This is deterministic engine code on the governed action path — never an agent decision, never a textual-claim trust, and **never a check-runner** (it consumes results; it does not execute checks).

Splitting these mirrors GitHub's "the runner executes; branch protection adjudicates" structure and is what makes the verdict **agent-non-forgeable for a fixed check definition**: the agent can influence neither the production (it does not run the executor) nor the adjudication (it does not write the ledger and the gate is deterministic). An agent writing its own `success` record — or the engine trusting an agent's "tests pass" — is the cardinal stub/fake-success sin this design exists to make impossible.

## The trust contrast with governance (why the ledger is hardened)

The governance PRD deliberately leaves its local review store (`local_review_verdicts`) **forgeable by direct DB write** (its R6, High) — an accepted leak, because human reviewers carry out-of-band accountability and the review is a _human's_ judgment. Actions cannot accept the analogous leak: a check result is a **machine's** verdict about whether the committed check ran and passed, produced for an agent that has **no reputation, job, or human accountability to lose by lying**. So the Actions ledger is the **hardened** counterpart — signed, agent-unwritable, deterministically recorded, SHA-bound — because the entity it protects against (a goal-directed agent optimizing to report "done") will forge the verdict the instant it is left forgeable. This is not over-engineering; it is the minimum that makes a machine-produced "the checks ran and passed" gate mean anything. (The v1 HMAC residual — shared OS user — is named honestly in scope; the engineering hardening to fully close agent forgery is OS-sandbox + Ed25519.)

| **The butler daemon/CLI**                         | The butler executor (trusted producer) | UC-EXEC-01/02 (runs the real check, executor≠agent), UC-EXEC-03 (injects minimal isolated secrets, masks output), UC-EXEC-05 (timeout/concurrency/observability), UC-LEDG-01 (signs the result)                                                  |
| **Claude Code** (coding harness)                  | Orchestrator                           | UC-GATE-03 (consumes the STEER signal, re-routes the agent — incl. running a stale check), UC-GATE-01 (bounded by the gate outcome)                                                                                                              |

Coverage check: every UC is primarily exercised by at least one persona; **GitButler (the engine)** is the deterministic actor behind every recording AC (LEDG) and every adjudication AC (GATE), and the **butler executor** is the producer behind every EXEC AC (and the pre-merge `on-merge-attempt` run) — every such AC names the recorder, the gate, or the executor as the WHO, never the gated agent.
