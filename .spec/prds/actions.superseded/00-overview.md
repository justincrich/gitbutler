# Actions — Configurable Agent-Non-Forgeable Validations for GitButler — Overview

> **Naming disambiguation (read first).** This **checks/validations** system is surfaced on the CLI as the **`but check`** noun — it is **not** the same feature as GitButler's existing "Actions" (the `butler_actions` macro/automation table that records agent tool-call automations). To avoid colliding with that established feature, the agent-facing CLI surface for *this* system is `but check {define, list, run, results, required}`, and the committed config lives at `.gitbutler/actions/*.toml`. Where this document says "a check" it means "a named check in that config"; it never means a `butler_actions` macro. The initiative codename remains "Actions" only because the capability is a GitHub-Actions analog; the shipped noun is `but check`.

## Product Description
This initiative turns **GitButler into a verification-enforcement layer for orchestrated multi-agent coding**. It introduces a GitHub-Actions-like, **config-as-code** system of **named checks/validations** (CLI noun **`but check`**) that run on a change and produce a **verifiable result** which a merge gate enforces. A check is a named validation — tests, lint, typecheck, or any custom command/script — defined in committed, ref-pinned config (`.gitbutler/actions/*.toml`), executed by a **butler-controlled executor that is not the agent**, and recorded as a **signed, SHA-bound, agent-unwritable** result. A change cannot land until **every required check reports a signed `success` bound to the current head SHA**.

### What the gate actually proves (and what it does not)
The guarantee is **precise and bounded**: the gate proves that **the committed, target-ref-pinned check definition actually ran by the trusted executor and exited `0` against the exact head SHA being landed** — it does **not** prove "the code is good" in any semantic sense. The check is only as strong as what the committed definition runs; a weak or vacuous check definition produces a weak guarantee. So the system is **non-fakeable BY THE AGENT for a FIXED check definition** — the gated agent cannot forge, replay, hand-edit, or self-produce a passing record for the checks the target-ref config pins — rather than "non-fakeable" in the unqualified, semantic-quality sense. The load-bearing protection is over *provenance and binding* (who ran the check, against what code, was the record authored by the trusted producer), not over *whether the check itself is a good test*. Because the strength rests entirely on the committed check definition, **the required-check set is self-protecting**: a change that adds, removes, or weakens a required check must itself clear the currently-required checks at the target ref before it can land (the **bootstrap-invariant** — see [UC-DEFN-04](./04-uc-defn.md#uc-defn-04-ref-pinned-definitions-read-at-the-target-ref-no-self-weakening) and [UC-GATE-05](./07-uc-gate.md#uc-gate-05-self-protecting-required-check-set-the-bootstrap-invariant)), the same self-escalation-prevention shape governance uses for its ref-pin.

Actions is the **verification/quality half** of "the way to production code is governance + accountability." The companion governance PRD (`.spec/prds/governance/`) enforces **process** — that a principal *may* perform an action and that an *approval exists* — but it does not enforce **quality**, and an agent's claim that "tests pass" is fakeable. Actions closes that gap: it makes "the agent-written code **provably ran the committed checks and they exited 0**" a property of the system rather than a hope, by gating the merge on **agent-non-forgeable, customizable validations** whose result the agent cannot produce, forge, replay, or hand-edit for a fixed check definition.

The enforcement surface is **GitButler's own action set** (`but` CLI / `but-api` / Tauri / N-API) — exactly the boundary the governance PRD already gates. Actions adds three things to that boundary: a **producer** (the executor, run in the trusted daemon/CLI the human runs, never in the agent's process), a **ledger** (deterministic, signed, agent-unwritable recording of `(name, head_sha, conclusion, producer-identity)`), and a **required-checks clause** on the merge gate (a deterministic consumer of the ledger). It is native to GitButler and grounded in GitButler's own crates and graph model.

## The thesis: the gate is a fact-check, not a trust exercise
> **You cannot ask the entity being graded to grade itself.** GitHub's branch protection works because the entity that is gated (the PR author) never produces the gating verdict — a trusted, separate principal (the App, bound to the head SHA) does. Whoever runs the check controls the result. If the gated agent also runs the gate on a surface it controls, the gate is theater.

A coding agent has **no out-of-band stakes** — no reputation, no job, no human accountability — so the usual human-accountability backstop that GitHub leans on does not exist. Prevention must therefore be **total and system-owned**, not advisory. Actions makes the gate a **deterministic fact-check** over signed records, never a trust exercise over an agent's textual assertion: the gate trusts only a record with a **valid producer signature bound to the current head SHA**, produced by an executor that **is not the agent**, recorded in a ledger the agent **cannot write to**, and **invalidated on every SHA-changing graph mutation**. This is the same no-stub / real-services / "subagents reward-hack 'tests pass'" discipline the repo already enforces, made into an engine guarantee. The fact it certifies is narrow and honest: *this committed check ran and exited 0 on this SHA* — not *this code is correct*.

This is the deliberate inverse of one accepted leak in the governance PRD: governance's local review store (`local_review_verdicts`) is **knowingly forgeable by direct DB write** (its R6, accepted because human reviewers carry accountability). The Actions ledger is the **hardened** counterpart — signed, agent-unwritable, deterministically recorded — because a machine-produced "this check ran and passed" verdict has no human behind it and so cannot be left forgeable by the agent.

## Problem Statement
Teams now run parallel coding agents against a real local git client, and the only act with lasting consequence — **landing code** — has no quality boundary the system enforces. Three structural gaps:

1. **Process governance is not quality governance.** The governance merge gate proves a principal holds `merge` and that an approving review exists at head. It proves **nothing about whether the code works** — it has no notion of "the tests ran and passed," "the typechecker is clean," or "the custom validation succeeded." A change can clear every governance clause while being broken. Governance itself names this: *"auto-run validation feeding the gate … required-checks-green is deferred; the POC's gate is review + permission only."* Actions is that deferred half.

2. **An agent's "it passes" is fakeable, and the agent has no stake in honesty.** An agent reports "tests pass" as a textual claim. It can fake success, stub the core logic, run a narrower test than required, or simply assert green. Nothing forces the validation to actually execute, nor binds the result to the exact code being landed. Unlike a human contributor, the agent has no reputation or accountability to lose by lying — so a verdict the agent can produce is worthless as a gate.

3. **A "passing" result is not bound to the code it claims to validate.** Even an honestly-produced result is meaningless if it can be **replayed from an ancestor commit**. GitButler rewrites the git graph constantly — rebases, commit edits, reorderings via the rebase editor — so a green produced against one SHA must not silently count for a different, later SHA. Without a hard **SHA-reset invariant**, an agent could pass a check once, then mutate the change underneath the green.

## Solution Summary
**Govern the quality of the landed code by making the merge gate consume an agent-non-forgeable check-result ledger.** Actions builds the producer, the ledger, and the consumer over GitButler's own action surface, composing with the existing governance gate. Four functional groups:

- **Action Definition** (DEFN): config-as-code **named checks** in committed, ref-pinned `.gitbutler/actions/*.toml` — each a `name` + a `trigger` (`on-commit` / `on-merge-attempt`) + a `run-spec` (a command or a repo `./path` script) + a `required` flag + a success mapping (process exit 0 → conclusion `success`). The check config is read at the **target ref**, exactly like governance's `gates.toml`, so an agent cannot weaken or remove a `required` check in its own change — and the required-check set is **self-protecting** (the bootstrap-invariant: a change to the required set must itself clear the currently-required checks). v1 is **local-action resolution only** (a command or a repo-local script); `uses`-registries, JS/Docker/composite action types, matrix/DAG/reusable workflows, and the `${{ }}` expression engine are deferred.

- **Check Executor** (EXEC): the butler-controlled **default executor** that runs a defined check against a checked-out head SHA in a clean workspace, and captures the conclusion. The executor runs in the **trusted daemon/CLI the human runs — NOT the agent's process** (executor ≠ the gated agent). It performs **real execution** — no stubbed success, ever — and injects only minimal, executor-isolated butler secrets the check needs. Separating who-executes from who-is-gated is the load-bearing anti-cheat: whoever runs the check controls the result, so the gated agent must never produce its own passing record.

- **Check-Result Ledger** (LEDG): **signed `(name, head_sha, conclusion, producer-identity)`** records on an **agent-unwritable** recording surface, written by **deterministic engine code** (recording a result is never an agent decision). The ledger enforces the **SHA-reset invariant** — a result is valid for the **current** head SHA only, and is **invalidated/recomputed on every graph/ref mutation**; a green bound to an ancestor SHA never counts. This is the contract the gate trusts: provenance over claims, bound to the exact commit, tamper-evident, and impossible for the agent to author.

- **Required-Checks Merge Gate** (GATE): a **required-checks clause** that **consumes** the ledger — it blocks a merge unless **every required check has a signed `success` bound to the current head SHA**. It **composes** with the governance merge gate (it is a new clause alongside `merge`-authority and the review requirement, evaluated by the same deterministic on-action code), **fails closed** on an unknown, missing, unreadable, or stale result, and emits the governance-consistent `{code, message, remediation_hint}` denial plus STEER fields so the orchestrator can **redirect** the agent to the failed/missing check. The gate is a deterministic consumer; **it never runs a check** and never trusts an agent's word.

The bet: hold agent-written code to a **provable** quality bar by making GitButler enforce, on its own merge action, that every required validation actually ran and exited 0 against the exact code being landed — produced by a trusted executor that is not the agent, recorded in a ledger the agent cannot forge, and reset the instant the code changes. The bar's *strength* is the committed check set's; the bar's *integrity* (it really ran, on this SHA, recorded by the producer) is the engine's.

## How Actions composes with governance + STEER
Actions is **not standalone** — it is the producer half of a three-part loop the governance PRD already established the consumer half of:

| Stage | System | Role | Mechanism |
|---|---|---|---|
| **Produce** | **Actions** (this PRD) | The agent-non-forgeable verification producer | Executor (≠ agent) runs the check → signs `(name, head_sha, conclusion)` → deterministic ledger write |
| **Consume / block** | Governance merge gate (extended) | The deterministic consumer | A new **required-checks clause** alongside the existing `merge`-authority + review clauses — block unless every required check is a signed `success` @ current head |
| **Redirect** | STEER | The corrective signal | On a missing/failed check, the denial's `remediation_hint` + STEER fields name the check and route the agent to fix it (incl. a stale-at-head check → "run the check") |

Together this is "the way to production code": Actions proves the committed checks ran and passed, the governance gate refuses to land the change otherwise, and STEER turns a red check into the agent's next action. Actions reuses governance's exact denial contract (`{code, message, remediation_hint}` + STEER fields) and its ref-pin discipline (config read at the target ref), and it strengthens governance's one knowingly-forgeable surface (the review store) with a signed, agent-unwritable ledger.

> **Honesty note (carried throughout).** Actions hardens the *verification verdict's provenance and binding*; it does not, by itself, close governance's accepted **fence** leak (raw `git` / `--no-verify` / a direct ungoverned N-API call can still land outside the governed `but` merge action), and it does not certify *semantic* code quality (only that the committed check ran and exited 0). The required-checks clause binds the **governed** merge, exactly as governance's review clause does. The executor isolation is a **clean-workspace** boundary, not a hostile-code sandbox in v1 (the personal-tenant / internal-agent trust model), and in that model the agent shares the OS user with the executor — so v1 HMAC raises the forgery bar but does not fully close agent forgery. All of these are named precisely in [01-scope.md § Known Limitations](./01-scope.md#known-limitations), never presented as boundaries.
