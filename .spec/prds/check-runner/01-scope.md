# Scope

**Scope Posture**: Full feature (a complete, polished v1 slice â€” deliberately scoped tight).

The boundary sentence for every item: **Check Runner makes a change unmergeable until every required check has a `success` bound to the current head OID, produced by a butler-controlled local runner that is not the agent, recorded in a plain store, and read mechanism-agnostically from the head commit â€” with a STEER denial on any miss.** It is the verification PRODUCER half; the governance merge gate is the CONSUMER (a new required-checks clause); STEER is the redirect. Everything opinionated about _which_ checks run lives in committed, ref-pinned config; the engine ships the mechanism only â€” a real runner, a plain result store, and a deterministic gate clause.

## The bar is "honest path is cheapest," not "cheating is impossible" (constraint, not a feature)

The threat model is **personal-tenant, own-fleet**: trusted human, trusted machine, an agent fleet the human runs. The goal is to make the **honest path** (let the runner run the check) **cheaper than cheating**, because a reward-hacking agent optimizes for least resistance. Two cheap things carry it: **reproducibility** (a forged green is **detectable** by re-running â€” though only _post-merge_, see Known Limitations; the honest path is cheap to take **for a fast, deterministic check**) and the **structural locks** (the gate reads a stored fact, not agent prose; runner â‰  agent; no caller-supplied-conclusion API; SHA-binding). v1 has **no runtime isolation and no cryptographic non-forgery layer** â€” and is consistent in this with the governance review clause, which already gates merges on a verdict store it accepts as forgeable (its R6). A check is another deterministic review that is **safer in one dimension** (reproducible â†’ post-merge lie-detection) but **weaker in another** (a check row carries no principal identity, so forgery isn't deterred by attribution the way impersonating a named reviewer is) â€” so its store needs no _more_ protection than the review store, but the asymmetry rests on reproducibility + the structural locks, **not** on accountability.

## The composition assumption (constraint, not a feature)

Check Runner does **not** re-invent the gate. The required-checks clause **extends the existing governance merge gate** (`.spec/prds/governance/`), evaluated as part of the same `enforce_merge_gate` code alongside the `merge`-authority and review clauses. It reuses governance's denial contract (`{code, message, remediation_hint}` + the STEER fields `class`/`held_permissions`/`authorized_actions`/`do_not`), its ref-pin discipline (config read at the target ref), and its fail-closed posture. This PRD specifies the producer (the runner), the contract (the plain result store), and the new clause (the gate). Two governance-side dependencies are named, not assumed: the STEER steering fields the denial reuses are delivered by governance **STEER-001** (sprint-07, **unmerged** â€” the GATE group sequences after it), and a **mechanism-agnostic local-merge entry point** must exist for the clause to gate a local `but merge` (the shipped `enforce_merge_gate` is forge-`review_id`-keyed â€” see [00-overview.md](./00-overview.md) and `08-technical-requirements/`).

## In Scope

### Check definition (config-as-code) â€” DEFN

- **Named checks** in committed, ref-pinned `.gitbutler/checks/*.toml`: each = a stable `name` (the analog of a GitHub status `context`) + a `trigger` + a `run-spec` + a `required` flag + a success mapping.
- **Triggers** restricted to two: `on-commit` (eligible when a commit is created) and `on-merge-attempt` (the trusted CLI/daemon runs the required checks immediately before a governed merge). No cron / dispatch / external event.
- **Run-spec**: a **command** string **or** a repo-local `./path` script â€” **local resolution only**, butler-run. No `uses` registry, no marketplace, no remote-ref / Docker / JS / composite resolution.
- **Success mapping**: process exit `0` â†’ conclusion `success`; non-zero â†’ `failure` â€” a deterministic function of the real run, never an agent's claim.
- **Required-set policy** (`[[required_check]]` in governance `gates.toml`, read at the target ref, mirroring the `[[gate]]` review requirement): names which defined checks are mandatory for a branch's merge gate. A defined check not in the required-set runs and records but does not block.
- **Ref-pinned at the target ref** (exactly like `gates.toml`), so a change cannot weaken, drop, or flip `required: false` on a required check **in its own head** to slip the gate; and the required-set is **self-protecting** â€” a change to it must itself clear the currently-required checks (the **bootstrap-invariant**).

### Check runner (the producer) â€” RUN

- A butler-controlled **local runner** that obtains a **clean checkout at the exact current head OID**, runs the defined `run-spec`, captures stdout/stderr + the exit code, and maps it to a typed conclusion.
- **Runner â‰  the gated agent**: it runs in the **trusted daemon/CLI the human runs**, NOT the agent's process â€” the load-bearing structural lock (whoever runs the check controls the result). It exposes **no path by which the agent supplies a conclusion**.
- **Mechanism-agnostic**: the checkout/binding works identically across GitButler **virtual branches**, **worktrees**, and **plain git** â€” it binds to the head _commit OID_, never assuming the live worktree == the head OID, and never disturbing or contending on the agent's shared worktree.
- **Real execution, no stubbed success â€” ever**: the runner actually invokes the command/script; a stubbed pass, a fake conclusion, or a skip-that-records-green is the cardinal sin, not a v1 simplification. A check it cannot run concludes **closed** (`failure`/`timed_out`), never `success`.
- **Result bound to the head OID**, recorded by deterministic engine code in a **plain `but-db` `check_results` table** (recording a result is never an agent decision). The result is keyed `(name, head_oid)`.
- **SHA-reset by construction**: a result is valid for its head OID only; any graph/ref mutation that moves the head OID leaves the new head with no satisfying result until the check is re-run â€” a green never carries across OIDs.
- **Timeout / concurrency / observability**: a check exceeding `timeout_secs` is terminated and concluded `timed_out`; concurrent runs key on `(name, head_oid)` and never clobber; a run emits an observable event with **dual-audience** output (human CLI + machine `--json`). (Checks that need butler-held credentials â€” secret injection â€” are **deferred**; v1 runs checks that need no secret. See Out of Scope.)

### Required-checks merge gate (the consumer) â€” GATE

- A **required-checks clause**: a change may merge into the target **iff** for **every** required check name (from the target-ref `[[required_check]]` policy) there is a **`success` bound to the current head OID** in `check_results`.
- **On-merge-attempt is a pre-merge run + read-only gate**: the trusted CLI/daemon runs the required `on-merge-attempt` checks immediately before the governed merge; the gate then consumes the current-head results. The gate itself **never runs a check** â€” a stale-at-head required check makes it BLOCK with `check_stale_at_head` + STEER ("run the check"), not auto-re-run.
- **Composes** with the governance clauses (same `enforce_merge_gate`, alongside `merge`-authority + review) â€” process _and_ quality in one decision.
- **Fail-closed**: a required check that is **missing**, **failed** (`failure`/`timed_out`/`cancelled`), **stale** (a result exists for the check but bound to a non-current head OID), or whose target-ref config is **unreadable/malformed** â‡’ **merge blocked** (`config.invalid` for a bad config), never vacuously satisfied. Enforced **independent of the branch's `protected` flag** â€” a branch carrying required checks is gated even if it is not in the governance _protected_ set (no fail-open via the protected-branch early-return).
- **STEER denial**: on a miss, the gate returns the governance-consistent `{code: "gate.check_required", message, remediation_hint}` + the steering fields â€” `class` (`ActorCorrectable` for a runnable miss; `OperatorRequired` for `config.invalid`), `authorized_actions` (e.g. `{command: "but check run <name> --head <oid>", effect: "produce the missing check result"}`), and `do_not` where applicable â€” distinguishing miss-reasons (`check_missing` / `check_failed` / `check_stale_at_head` / `config.invalid`). Dual-audience (`--json` for the orchestrator).
- **Bootstrap-invariant**: a change that edits the required-set or a required check's definition must itself clear the currently-required checks at the target ref before it can land.
- **Mechanism-agnostic gate entry**: the clause must be reachable from a **local `but merge`** (virtual-branch / worktree / plain-git, no forge PR) that resolves `(source_ref, target_ref, head_oid)` **without a forge `review_id`**. The shipped `enforce_merge_gate` is currently forge-review-keyed, so generalizing its entry (head OID via a mechanism-agnostic `gix` ref-peel) is a v1 requirement, not an assumption.

### Human-facing UI (desktop, v1.1) â€” UI

v1.0 is CLI-first; **v1.1 un-defers a focused desktop result-viewing surface** (grounded in a `frontend-designer` review of the real `apps/desktop/` UI â€” see [`08-technical-requirements/10-frontend-ui.md`](./08-technical-requirements/10-frontend-ui.md)). GitButler today renders forge checks only as a single aggregate `CIChecksBadge` that links out to GitHub; the per-check `CiCheck[]` data is fetched and **cached** (the `listCiChecks` query caches `CiCheck[]`), but every current consumer reads it through `ChecksMonitor`'s `parseChecks` call-site transform that collapses it to an aggregate â€” so nothing renders the per-check detail today. In scope for **v1.1**:

- A **per-head `CheckResultsPanel`** (the visual form of `but check results`, UC-RUN-05) **+ a "Checks" segment** on the existing `[projectId]/branches` explorer (real segments today: `All / PRs / Local` from `BRANCH_FILTER_OPTIONS = ["all","pullRequest","local"]`) â€” a **state of an existing route, not a new route**. Net-new: `CheckConclusionBadge` + `CheckResultRow` + `CheckResultsPanel`, all composing existing `packages/ui` primitives. **Modest, not free**: the panel requires a query-path change (a parallel `listCiChecks` without the `parseChecks` transform, or a `ChecksMonitor` refactor) and type-system surgery to add the "Checks" segment (extend `BRANCH_FILTER_OPTIONS`, the `BranchFilterOption` union, `isBranchFilterOption`, the `filterOptions` block in `BranchExplorer.svelte`, and a new `{type:"checks";headOid}` arm in the `BranchesSelection` union in `BranchesView.svelte`).
- Deferred (named): the **merge-gate summary** (UC-GATE â€” blocked on governance's own deferred merge dialog, and sequentially on Surface 1 above), the **Checks settings tab** (UC-DEFN â€” needs the DEFN CLI first; also requires governance settings tab wiring to be confirmed), the **cross-branch checks matrix** (needs a `but check results --all-branches` capability), and any **lite (React) checks UI** (no checks UI exists in `apps/lite/`; `packages/ui` is Svelte-only â†’ a **port**, not a share).

## Out of Scope

### Deferred (named as out-of-threat-model, NOT owed debt)

- **Cryptographic non-forgery** â€” signing as security, an agent-unwritable hardened ledger, HMAC/Ed25519, an OS-sandboxed runner. Not needed under the own-fleet reproducibility bar; a nullable `signature` column is retained only as a forward-compat seam for off-host producers, explicitly not a v1 security claim. _If_ the model later changes (untrusted contributors, remote/multi-machine producers), that seam is where provenance-signing earns its place.
- **Untrusted / fork-contribution execution** â€” running a hostile contributor's check safely (the `pull_request` vs `pull_request_target` split, ephemeral hostile-code sandboxing). v1 runs the repo's own committed, trusted check definitions only.
- **A detached/remote runner protocol** â€” lease / long-poll / labels / runner-groups / autoscaling. v1 runs synchronously in the trusted local daemon/CLI.
- **Action types beyond local** â€” JavaScript / Docker / composite action types, a `uses` registry / marketplace, remote-ref (`owner/repo@sha`) resolution. v1 is a command or a repo `./path` script.
- **Workflow orchestration** â€” matrix / strategy, a job DAG (`needs`), reusable workflows. v1 is a flat list of named checks.
- **A full expression engine** â€” the `${{ }}` contexts/functions. v1 maps an exit code to a conclusion.
- **Merge-queue / "require up to date"** â€” serialized landing + "head must be at/after target tip." v1 evaluates required checks at the current head; the gate-eval-vs-merge race is named in Known Limitations.
- **Consuming third-party forge CI as a first-class required producer** â€” `ci_checks` already caches forge check-runs; wiring it in as a second producer behind the same clause is the natural next layer (the `check_results` contract is designed to admit it), but v1 ships the local runner as the sole producer.
- **Secret injection for checks that need credentials** â€” a `secrets`/`env` declaration on a check + scoped injection + output masking. v1 runs checks that need no butler-held secret (tests / lint / typecheck); a check needing a private-registry token etc. is deferred (and would re-introduce the masking/runtime-exfiltration caveats GitHub also carries).
- **Rich result UX** â€” annotations (file+line), live log-streaming, a results dashboard. v1 captures the gate-critical conclusion + a log link.
- **The deferred UI surfaces** â€” the merge-gate summary (blocked on governance's deferred merge dialog), the Checks settings tab (needs the DEFN CLI first), the cross-branch checks matrix, and any **lite (React)** checks UI (Svelte-only library â†’ a port). The v1.1 result-viewing surface (per-head panel + branches "Checks" state) is now **in scope** (above); the agent-facing surface stays CLI (`but check â€¦`).

### Genuinely out (non-goals)

- **Certifying semantic code quality** â€” the gate proves the committed check ran and exited 0, never that the code is _correct_ or the check is a _good_ test.
- **Driving the agent's reasoning** â€” Check Runner governs _whether a validated change may land_, never the agent's prompts or turn loop (the harness owns the agent).
- **Closing governance's fence leak** â€” raw `git` / `--no-verify` / an ungoverned direct N-API call still bypass the governed `but` merge (the same accepted-leak class as governance R1/R14). The clause binds the **governed** merge only.
