# Roles

This initiative draws one decisive boundary: between the entity that **is gated** (the coding agent whose change must be proven to have passed its checks) and the entity that **produces the verdict** (the butler runner the human runs). The anti-cheat rests on those never being the same principal — _whoever runs the check controls the result._ Every role is described by **what it may produce, record, and adjudicate**, because that — plus the reproducibility of a check — is what makes the honest path cheaper than cheating.

| Role                                                     | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Human fleet-owner / admin**                            | The human who **defines checks** (authors `.gitbutler/checks/*.toml` — `but check define` writes/validates the file; hand-editing also works), **marks checks `required`** (in `gates.toml`'s `[[required_check]]`), and runs the trusted daemon/CLI the runner executes in. In the governance composition they are the maintainer/administrator (`merge` + `administration:write`). The `define`/edit step writes an **uncommitted** file (ungated); only the **committed change landed at the target ref** is governed — it is ref-pinned, the merge that lands it is gated, and a change to the **required-set** must itself clear the currently-required checks (the bootstrap-invariant). They decide _which_ validations define "good code"; the engine enforces _that_ those validations actually ran and passed (not _that_ the code is semantically good). The runner runs as them, on their machine — the agent cannot reach this root.                                                                                                                                                                                                                 |
| **Implementer / reviewer agents (the gated principals)** | The LLM workers the orchestrator runs — the **subjects of checks**. They produce the _code_ a check validates, but they **cannot record, or claim their way past,** a required check: the gate never reads their prose, only a stored result the runner produced at the current head. An implementer cannot land a change until its required checks are `success` at head; a denied agent receives the `{code, message, remediation_hint}` + STEER fields naming the failed/missing check and the corrective `but check run …` so it can fix and re-run. The agent runs `but` actions; it does **not** run the runner and there is **no public path by which it supplies a conclusion**. (Threat model named in scope: the agent shares the OS user with the runner, so it could in principle write the plain `check_results` table directly — a forged row the gate accepts **at merge time**, **detected only by a later re-run, post-merge**. Running the check is cheaper than forging **for a fast, deterministic check**; for slow/flaky checks the deterrent rests on the structural locks, not cost. See [01-scope.md](./01-scope.md) Known Limitations.) |
| **The butler runner (the trusted producer)**             | Not a principal the agent can assume — the **butler-controlled process that runs checks.** It runs in the **trusted daemon/CLI the human runs**, obtains a **clean checkout of the current head OID** mechanism-agnostically (virtual branch / worktree / plain git) **without disturbing the agent's shared worktree**, **actually executes** the defined command/script (no stubbed success), enforces `timeout_secs` (a timeout → `timed_out`, never a hang or a silent green), runs/records concurrently without clobbering (`(name, head_oid)`-keyed), captures the exit code → conclusion, and binds the conclusion to the head OID it ran against. It is also what runs the required `on-merge-attempt` checks **immediately before** a governed merge (the pre-merge run), so the gate has current-head results to consume. It **produces** the inputs; it does **not** adjudicate the gate. **Runner ≠ the gated agent** is the single most important fact in this PRD.                                                                                                                                                                                  |
| **GitButler engine (the deterministic recorder + gate)** | Not a principal — the policy-enforcement system, split across two deterministic responsibilities. As the **recorder**, it writes the result to the plain `check_results` table as engine code (never an agent decision), keyed by head OID so a mutation that moves the head leaves the new head unsatisfied. As the **gate**, it evaluates the required-checks clause at the governed merge boundary — reading `check_results` for the **current head OID**, requiring `success` for every required check, and blocking otherwise — composing with the governance `merge`-authority and review clauses. The gate is a **pure read-only consumer**: it **never runs a check** (a stale-at-head check makes it BLOCK with `check_stale_at_head` + STEER "run the check"), never accepts a caller-supplied conclusion, and never trusts an agent's claim. It enforces the **bootstrap-invariant** and the gate runs **independent of the `protected` flag** for any branch carrying a required-set.                                                                                                                                                                 |
| **The orchestrator (coding harness)**                    | An external harness (Claude Code, Codex, a ralph loop) that dispatches the agents and drives GitButler via the `but` CLI. **Trusted by outcomes, not policed** (same trust class as governance — your machine, your keys, your loop). It is bound only by the gate's outcome: it cannot land a change whose required checks are not green at head. It **consumes the STEER signal** — on a denial, it re-routes the agent to run/fix the named check (including running a stale-at-head check via `but check run`). It does not produce results and is not the runner; under the pre-merge step it is the trusted process that invokes `but check run` before calling merge.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

## The producer-vs-adjudicator separation (the load-bearing split)

GitButler's deterministic-vs-probabilistic doctrine maps onto two distinct deterministic responsibilities, deliberately **not** the same component:

- **Produce** (the runner): obtains the head-OID checkout, runs the real check, captures the real conclusion, records it bound to the head OID. Butler-controlled code, never the gated agent. Also runs the `on-merge-attempt` checks in the pre-merge step.
- **Adjudicate** (the engine gate): reads `check_results` for the current head, requires `success` for every required check, decides merge / block. Deterministic engine code on the governed action path — never an agent decision, never a textual-claim trust, and **never a check-runner** (it consumes results; it does not execute checks).

Splitting these mirrors GitHub's "the runner executes; branch protection adjudicates," and it is what keeps the verdict trustworthy _without crypto_: the agent influences neither the production (it does not run the runner) nor the adjudication (the gate is deterministic and reads only stored results). A green it forges directly into the plain store is **not undone** — re-running **detects** it (post-merge); it does not auto-revert the merge. The deterrent is detectability + the structural locks, not reversal.

## Why this needs no hardened ledger (the contrast with governance)

Governance leaves its human-review store (`local_review_verdicts`) **forgeable by direct DB write** (its R6, accepted because a human reviewer carries out-of-band accountability and a review is a _judgment_ that cannot be re-derived). A check differs on **two** axes, and honestly only one favors the check: it is **reproducible** (so a forged green is _detectable_ by re-running — though only **post-merge**, not at the gate), but it also carries **no principal identity** (forging a check row needs only file-write, with no named reviewer to impersonate — so it **loses** the _attribution_ deterrent the human verdict has). So Check Runner does **not** need the hardened, signed, agent-unwritable ledger the `actions` PRD proposed — but the reason is **not** "reproducible therefore strictly safer." It is that, under the **own-fleet threat model**, the cheap structural locks (runner ≠ agent, gate-reads-stored-fact, no-conclusion-API, SHA-binding) plus post-merge detectability are **enough** — the same bar governance already accepted for its forgeable review store. The plain `check_results` table is the _right_ design under that threat model (not a compromise), with the residual forgery window named honestly in [01-scope.md](./01-scope.md) Known Limitations.

| **Justin** (fleet manager / governance owner)     | Human fleet-owner / admin            | UC-DEFN-01/03 (defines named checks, marks `required`), UC-DEFN-04/05 (ref-pins + bootstrap-invariant), UC-GATE-02 (configures which checks gate the merge) |
| **An implementer** (`contents:write`, no `merge`) | Implementer agent (gated principal)  | UC-RUN-01 (its commit triggers the runner), UC-GATE-01 (its merge is blocked until checks green at head), UC-GATE-03 (receives the STEER redirect)          |
| **A reviewer** (`reviews:write`)                  | Reviewer agent (gated principal)     | UC-RUN-03 (cannot author a passing record), UC-GATE-01 (the check result it cannot fake is what gates)                                                      |
| **The butler daemon/CLI**                         | The butler runner (trusted producer) | UC-RUN-01/02 (runs the real check at the head OID, mechanism-agnostic), UC-RUN-03/05 (records, timeout/concurrency), UC-RUN-04 (pre-merge run)              |
| **Claude Code** (coding harness)                  | Orchestrator                         | UC-GATE-03 (consumes STEER, re-routes the agent), UC-GATE-01 (bounded by the gate outcome)                                                                  |

Coverage check: every UC is primarily exercised by at least one persona; the **engine** is the deterministic actor behind every recording (RUN-03) and adjudication (GATE) AC, and the **butler runner** is the producer behind every RUN AC (and the pre-merge run) — every such AC names the recorder, the gate, or the runner as the WHO, never the gated agent.
