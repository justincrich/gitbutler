### M4 — STEER-003 "own-branch" determination mechanism for self-approve exclusion is underspecified

**Severity:** MEDIUM · **Type:** Authz correctness ambiguity · **Agents:** security-auditor (S-3); **contested by rust-reviewer**
**Location:** `STEER-003-gate-state-aware-authorized-actions.md:55` (MUST exclude `but review approve` for "CALLER'S OWN BRANCH"); STEER-003:193 (derivation signature)

**Evidence (contested):** The security-auditor flags that the derivation signature `authorized_actions(principal, denied={route_d, predicate_d, ref_d}, cfg)` has no field identifying branch ownership, and no AC tests the not-own-branch case. The rust-reviewer claims PASS at `menu.rs:232+` with "self-approve exclusions" verified — but did not show the mechanism.

**Risk:** If the implementer makes the exclusion unconditional (exclude `approve` for ALL commit-path denials), it's safe but over-restrictive (a reviewer who legitimately can approve another's work never sees the affordance). If conditional on an undefined "own-branch" check, the implementer might guess wrong.

**Expected:** STEER-003 must either (a) state explicitly that `approve` is excluded unconditionally for commit-path denials (conservative by design) and add a negative AC for the not-own-branch case proving the over-restriction is intentional, OR (b) define the own-branch comparison mechanism (e.g., commit-author equality, ref-name pattern) and add a positive AC for the not-own-branch case where `approve` IS offered.

**Fix:** Add a STRICTLY constraint or AC disambiguating the mechanism. The C5 subtraction is the load-bearing security property; the self-approve carve-out rests on top of it and must be precisely defined.

### M5 — STEER-009 does not replay the admin-write menu (4th menu-bearing denial type)

**Severity:** MEDIUM · **Type:** No-lying-menu coverage gap · **Agents:** security-auditor (S-8)
**Location:** `STEER-009-governed-loop-no-lying-menu.md:51` (MUST replays three denial types); `STEER-005` AC-4 (adds 4th menu-bearing type via `governance_cli_error`)

**Evidence:** SA-2's cycle-1 remediation expanded STEER-009 from "replay only `branch.protected`" to "replay `branch.protected` + `gate.review_required` + `perm.denied`" — three denial types. RR-2 (also cycle-1) added a fourth menu-bearing denial type: admin-write via `governance_cli_error`. STEER-009's MUST list was not updated to include the fourth type. The admin-write affordance row (read/inspect, request-config-change, discovery) is offered but never replayed to prove each entry actually succeeds for the denied caller.

**Risk:** If an admin-write affordance entry is non-runnable in its stated context (e.g. requires a different authority than `administration:read`), the no-lying-menu invariant is violated for that denial type. Likelihood is LOW (the affordances are read-only and likely trivially runnable), but the §9.1 invariant says "every `authorized_actions` entry ... never reproduces the (route, predicate, ref) that just failed" — "every" must be proven, not assumed.

**Expected:** Add a STEER-009 AC replaying the admin-write menu: for each `authorized_actions` entry in an `administration:write` denial, run it in its stated context and assert exit 0 or a legitimate non-`perm.denied` gate.

**Fix:** Either extend STEER-009 AC-1's scope, or add a new AC-4a. Alternatively document that admin-write affordances are trivially non-lying (all read-only, no gate) and add a lightweight assertion proving that property structurally.

### M6 — STEER-004 exhaustive-class match may be bypassed by direct constructor field assignment

**Severity:** MEDIUM · **Type:** Architectural ambiguity · **Agents:** security-auditor (S-6); **weakly contested**
**Location:** `STEER-004-class-mapping-branch-protected-wiring.md:56-57` (MUST exhaustive match + MUST populate `class=ActorCorrectable` on `missing_permission`)

**Evidence:** STEER-004:56 requires a `DenialCause` enum + exhaustive `match cause` with no `_ =>` arm (verified present at `authorize.rs:91-98`). STEER-004:57 says "MUST populate `held_permissions` + `authorized_actions` + `class=ActorCorrectable` on `missing_permission`" — this reads as direct field assignment in the constructor. If the constructor sets `class` directly rather than calling the classification match, the match's exhaustiveness is circumvented for that path. STEER-010 AC-3's `trybuild` (per M2, also missing) would check the match function, not the constructors.

**Risk:** A future `DenialCause` variant added without updating a hard-coded constructor's `class` assignment would compile silently (the match catches the variant, the constructor doesn't reference `DenialCause`) and ship with a stale `class` — silently miscategorizing a denial as `actor_correctable` when it should be `operator_required` (the security HIGH #2 failure mode).

**Expected:** Either (a) all constructors must route through the classification match (the match IS the only path to set `class`), proven by a grep asserting no direct `class:` field assignment outside the match, or (b) `class` becomes a computed property of `DenialCause`, not a settable field.

**Fix:** Add a STEER-010 grep AC for direct `class:` assignment in constructor code, OR restructure `class` as a method on `DenialCause`. The latter is the cleaner architectural fix.
